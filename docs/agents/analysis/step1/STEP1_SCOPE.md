# STEP1. 분석 범위 정의

목적: 분석 대상과 분석 타입을 먼저 고정해 이후 단계가 추정으로 흐르지 않게 한다.

## 진입 조건

- 사용자 요청이 있다.
- 아래 중 하나 이상을 식별할 수 있다.
  - 기능명
  - 화면명
  - 클래스명
  - 파일명
  - 오류 문구
  - 문서명
  - 사용자 행동
  - 관찰된 증상
  - 로그 키워드
  - 실행 환경 단서

## 행동 원칙

- 직접 열어 확인한 코드, 문서, 실행 결과만 사실로 기록한다.
- 확인하지 않은 내용을 확인한 것처럼 서술하지 않는다.
- 근거 없이 결론을 도출하지 않는다. 판단 근거가 없으면 이슈에 기록하고 사용자에게 확인을 요청한다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. 요청이 analysis 워크플로우 범위(현재 상태·동작·구조의 파악과 보고)인지 판정한다.
  → 범위 밖(코드 수정·구현·리팩토링·코드 리뷰): 강제 중단 조건 적용.
  → 범위 내: 2번으로 진행한다.

- [ ] 2. 생략어·맥락 의존 표현("그 버그", "아까 그 화면")을 직전 대화 맥락 또는 발화 내 단서로 복원한다.
  → 복원 가능: 복원 결과를 진입점 식별에 사용후 정리해 사용자에게 확인을 받는다.
  → 복원 불가: 이슈에 기록하고 사용자에게 질문한다.

- [ ] 3. 분석 타입을 분류하고 선정 이유를 기록한다. 두 타입 이상 해당 시 주/보조를 지정한다.
  - `code-flow`: 코드 호출 흐름, 상태 동작, 버그 원인 분석
  - `doc-consistency`: 문서 간 기준 충돌, 누락, source of truth 분석
  - `dependency-env`: SDK, 빌드 도구, 라이브러리, 개발 환경 분석
  - `ui-spec`: 화면 구조, 상태, 사용자 이해 가능성 분석
  - `product-scope`: 제품 범위, 정책, 단계 구분 분석
  - 주/보조 기준: 분석 결론에 더 직접 영향을 주는 타입이 주 타입, 결론을 보강하는 타입이 보조 타입.
  - 세 타입 이상 또는 경계 불명확: 사용자에게 먼저 질문한다.
  - 5가지 타입 중 0개 해당: analysis.md 분기 C 적용, 강제 중단

- [ ] 4. 진입점 정보(기능명, 화면명, 클래스명, 파일명, 오류 문구, 문서명)를 목록화하고 대상 기능/모듈을 확정한다.
  - 진입점 정보가 전혀 없으면: 강제 중단 조건 적용.

- [ ] 5. 주 타입 기준으로 핵심 파일을 선정한다.
  - 결론에 직접 영향을 주는 파일만 포함한다. 로직 없는 래퍼·단순 데이터 홀더·자동 생성 파일은 제외한다.
  - 0개이면: 강제 중단 조건 적용.

- [ ] 6. 보조 타입이 있으면 같은 기준으로 보조 타입 핵심 파일을 별도 선정한다. (해당 시)
  - 0개이면: 보조 타입 지정을 해제하거나 사용자에게 확인을 요청한다.

- [ ] 7. 선정된 각 파일에 선정 이유와 (주)/(보조) 표시를 붙이고, 제외 범위/시나리오를 명시한다.

- [ ] 8. STEP3(영향 범위) 적용 여부를 판단하고 기록한다.
  - 필수: 주 또는 보조 타입이 code-flow·doc-consistency·dependency-env 중 하나라도 해당하면.
  - 생략: 주·보조 타입이 모두 ui-spec·product-scope만 있으면.

- [ ] 9. code-flow 또는 dependency-env 타입이면 입력 fixture/재현 조건을 기록한다. (해당 시)
  - 없으면 "없음(해당 없음)" 또는 "없음(미확인)" 중 해당하는 것으로 명시한다.
    - 없음(해당 없음): 분석 결론이 재현 조건과 무관한 경우
    - 없음(미확인): 분석에 필요하나 아직 확보하지 못한 경우

- [ ] 10. dependency-env 타입이면 분석 환경(OS, SDK, 빌드 환경 등)을 기록한다. 없으면 "없음". (해당 시)

- [ ] 11. 의도·범위·기대 결과가 모호한 항목이 남아 있는지 최종 점검한다.
  → 진입점 정보가 전혀 없음: 강제 중단 조건 적용.
  → 모호 항목이 있음: 이슈에 기록하고 사용자에게 질문한다.
  → 없음: 계속 진행한다.

- [ ] 12. 산출물 형식에 맞춰 출력한다.

- [ ] 13. 진행도를 최종 갱신한다.

## 하네스

- [ ] 1번 통과 — analysis 범위 여부가 판정됐고, 범위 밖이면 강제 중단이 적용됐다.
     **CRITICAL** / 미통과 시: 1번 재수행. 범위 밖을 진행한 경우 즉시 중단.
- [ ] 2번 통과 — 생략어·맥락 표현 복원이 시도됐고, 복원 불가 항목은 이슈에 기록됐다.
     **HIGH** / 미통과 시: 2번 재수행.
- [ ] 3번 통과 — 분석 타입이 분류됐고 선정 이유가 기록됐다. 두 타입 해당 시 주/보조가 지정됐다.
     **HIGH** / 미통과 시: 3번 재수행. 경계 불명확이면 사용자 질문 후 재수행.
- [ ] 4번 통과 — 진입점 정보가 목록화됐고 대상 기능/모듈이 확정됐다.
     **HIGH** / 미통과 시: 진입점 정보 전혀 없음 → 강제 중단. 일부 누락 → 4번 재수행.
- [ ] 5번 통과 — 주 타입 기준 핵심 파일이 1개 이상 선정됐다.
     **CRITICAL** / 미통과 시: 0개이면 강제 중단. 선정 기준 오류이면 5번 재수행.
- [ ] 6번 통과 — 보조 타입이 있으면 보조 타입 핵심 파일이 선정됐다. (해당 시)
     **HIGH** / 미통과 시: 6번 재수행.
- [ ] 7번 통과 — 모든 선정 파일에 선정 이유와 (주)/(보조) 표시가 있고, 제외 범위가 명시됐다.
     **MEDIUM** / 미통과 시: 7번 재수행.
- [ ] 8번 통과 — STEP3 적용 여부가 "필수" 또는 "생략"으로 기록됐고 근거가 있다.
     **MEDIUM** / 미통과 시: 8번 재수행.
- [ ] 9번 통과 — code-flow·dependency-env 타입이면 fixture가 기록됐거나 "없음(해당 없음/미확인)"으로 명시됐다. 해당 없으면 자동 통과. (해당 시)
     **MEDIUM** / 미통과 시: 9번 재수행.
- [ ] 10번 통과 — dependency-env 타입이면 분석 환경이 기록됐거나 "없음"으로 명시됐다. 해당 없으면 자동 통과. (해당 시)
     **MEDIUM** / 미통과 시: 10번 재수행.
- [ ] 11번 통과 — 모호 항목이 이슈에 기록됐거나 "없음"이 확인됐다.
     **HIGH** / 미통과 시: 11번 재수행.
- [ ] 12번 통과 — 산출물이 형식에 맞춰 출력됐다.
     **HIGH** / 미통과 시: 12번 재수행.
- [ ] 13번 통과 — 진행도가 최종 갱신됐다.
     **LOW** / 미통과 시: 13번 재수행.

미통과 항목이 있으면 실패 규격에 따라 기록 후 강제 중단 조건을 확인한다. 루프 판정 기준은 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 참조

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 이슈/리스크에 기록한다.

## 산출물(STEP 2에 전달 정보)

STEP 완료 시 아래 형식으로 값을 채워 출력한다.

```text
[analysis workflow — STEP1 산출물]
타입     : analysis — code-flow / doc-consistency / dependency-env / ui-spec / product-scope 중 하나 — 선정 이유
           보조 타입이 있으면 "주: XX / 보조: YY" 형식으로 기록
단계     : STEP 1 (분석 범위 정의)
진행도   : (n/N) - 마지막으로 완료한 체크리스트 항목명 (진행중 / 완료)
           N = 실행 체크리스트 13개 중 실제 적용 항목 수.
           조건부 항목(6·9·10번)은 해당 없으면 "해당 없음"으로 처리하고 완료 수에 포함한다.

복원 매핑              : 원문 표현 → 복원 결과. 없으면 "없음"           → STEP2 진입점 보강
대상 기능/모듈         : 기능명 또는 화면명 — 요청에서 식별한 대상       → STEP2 진입점
핵심 파일              : (주) 파일명 — 선정 이유 (중요도 순)              → STEP2 파일 열기 기준
                         (보조) 파일명 — 선정 이유. 보조 타입 없으면 "없음"
제외 범위/시나리오     : 파일/모듈명 — 제외 이유. 없으면 "없음"           → STEP2 추적 경계
STEP3 적용 여부        : 필수 — 해당 조건 / 생략 — 이유                   → STEP3 진입 기준
입력 fixture/재현 조건 : 재현 조건 또는 입력값. 없으면 "없음(해당 없음/미확인)" → STEP2 (해당 시)
분석 환경              : OS, SDK, 빌드 환경 등. 없으면 "없음"             → STEP2 (해당 시)

이슈 : [범위/타입/파일/모호/기타] 내용 — 이유. 없으면 "없음"
```

→ 출력 예시는 [STEP1_SAMPLES.md](STEP1_SAMPLES.md) 참조

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `타입`이 요청의 실제 성격과 맞는가 — 다른 분석 타입이 더 적합하지 않은가
- [ ] `핵심 파일`이 실제 요청 기능과 직접 연결되는가 — 관련 없는 파일이 포함되거나 핵심 파일이 빠지지 않았는가
- [ ] `STEP3 적용 여부`의 판단 근거가 실제 분석 성격과 맞는가
- [ ] `이슈`가 "없음"이 아니면, 결론을 크게 흔들 가능성이 있는가 — 흔든다면 진행 여부를 직접 결정한다

```text
전달 데이터 : 타입 / 대상 기능/모듈 / 핵심 파일 / 제외 범위/시나리오 / STEP3 적용 여부 / fixture(해당 시) / 분석 환경(해당 시) / 이슈. 미완성 항목은 "미완성"
```

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- 요청이 analysis 범위 밖이다 → 사용자에게 적절한 워크플로우 안내 후 중단
- 타입 분류가 불가능하고 사용자 질문 후에도 해소되지 않는다 → 작업 중단
- 진입점 정보가 하나도 없다 → 사용자에게 범위 재정의 요청
- 핵심 파일이 0개다 → 사용자에게 진입점 정보 재확인 요청
- 모호성이 해소되지 않는다 → 사용자 응답 대기
- 루프 판정 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용
