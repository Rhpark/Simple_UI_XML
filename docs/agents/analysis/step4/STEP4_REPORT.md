# STEP4. 결과 보고

목적: 분석 결과를 사실과 추정으로 분리하고, 사용자가 바로 판단할 수 있도록 결론, 점수, 근거 범위, 우선 행동을 함께 보고한다.

## 진입 조건

아래 두 경로 중 하나를 통과했을 때 진입한다.

- 경로 A: STEP3 하네스를 통과했다.
- 경로 B: STEP1에서 `STEP3 적용 여부 = 생략`으로 판정됐고, STEP2 하네스와 [STEP4 직행 보강] 블록을 통과했다.

아래 항목이 실제 값으로 채워진 상태로 전달된다.
  - [ ] `타입` — 분석 타입과 선정 이유가 있다
  - [ ] `근거 파일` — 읽은 근거가 파일명:라인번호 또는 공식 URL로 기록돼 있다
  - [ ] `직접 파일` — 분석 결론 또는 후속 조치의 직접 대상이 기록돼 있다
  - [ ] `간접 파일` — 직접 파일의 결론 또는 후속 조치로 함께 영향을 받는 대상이 기록돼 있다. 없으면 "없음"으로 확인돼 있다
  - [ ] `사용자 영향` — 사용자에게 보이는 영향이 기록돼 있다. 없으면 "사용자 영향 없음"으로 확인돼 있다

## 실행 체크리스트

- [ ] 보고서 맨 앞에 한 줄 결론을 쓴다.
- [ ] `Agent 수행성`과 `사용자 이해성`을 각각 100점 기준으로 평가한다.
- [ ] 읽은 범위, 제외한 범위, 미확인 범위를 `범위`에 분리한다.
- [ ] 85점 미만 항목이 있으면 `우선 행동`에 우선 개선 행동을 기록한다.
- [ ] `근거 파일`과 `영향 계층` 등 이전 STEP 산출물을 근거로 확인된 사실과 추정/가정을 분리해 각각 명시한다. STEP3 생략 경로에서는 STEP2의 `문서 주장`, `분기 분석`, `[STEP4 직행 보강]`을 영향 계층 대신 사용한다. 판단 기준: 파일명:라인번호 또는 공식 문서 URL 근거가 있으면 사실, 근거 없이 추론했거나 읽지 않은 파일에 대한 판단이면 추정으로 분류한다.
  → 작성 후 사실 항목 각각에 파일명:라인번호 또는 공식 URL이 있는지 재확인한다. 근거가 없는 항목은 추정으로 이동한다.
- [ ] 원인 후보가 여러 개면 아래 기준 순서로 우선순위를 정한다.
  - 1순위: 사용자에게 직접 노출되는 동작 오류, 데이터 손실, 화면 멈춤
  - 2순위: 다른 STEP, 기능, 문서 기준까지 파급되는 원인
  - 3순위: 수정 후 테스트 영향이 큰 원인
- [ ] 다음 권장 단계는 review / feature / refactor 중에서 선택하고, 근거와 함께 기록한다. 복합이면 주/보조를 구분한다. 주/보조 판정 기준: CRITICAL 또는 HIGH finding이 있으면 해당 finding의 워크플로우가 주, 나머지는 보조다. 동일 등급에 여러 워크플로우가 섞이면 사용자 영향이 더 직접적인 쪽을 주로 선택한다.
- [ ] 각 finding에는 심각도, 근거 항목, 다음 행동을 함께 기록한다.
- [ ] CRITICAL finding이 있으면 현재까지 완성된 산출물을 먼저 출력하고, CRITICAL finding을 별도 블록으로 사용자에게 보고한다.
- [ ] 산출물 전체를 `docs/agents/output/analysis/YYYYMMDD_HHmmss.md` 형식으로 저장한다. (날짜·시각은 현재 기준으로 채운다)

## 하네스

- [ ] `한 줄 결론`이 작성됐다.
- [ ] `점수`에 Agent 수행성, 사용자 이해성 점수가 모두 있다.
- [ ] `감점 이유`에 각 점수의 감점 이유 또는 "없음"이 기록됐다.
- [ ] `범위`에 읽은 범위, 제외 범위, 미확인 범위가 분리됐다.
- [ ] `사실`과 `추정`이 분리됐다.
- [ ] `사실`의 각 항목에 파일명:라인번호 또는 공식 URL이 있다. 근거 없는 항목이 있으면 미통과.
     **CRITICAL** / 미통과 시: 근거 없는 사실 항목을 추정으로 이동 후 재수행.
- [ ] 진입 경로가 STEP3 통과인지 STEP3 생략 직행인지 확인됐다.
- [ ] 원인 후보가 2개 이상이면 우선순위가 명시됐다.
- [ ] `다음 단계`가 review / feature / refactor 중 하나 또는 주/보조 조합으로 기록됐다.
- [ ] 모든 finding에 심각도가 있고, 근거에 파일명:라인번호 또는 공식 URL이 있으며, 다음 행동이 있다.
- [ ] 산출물이 docs/agents/output/analysis/ 에 저장됐다.
     저장 경로: [경로를 여기 기재 — 비어있으면 미통과]
     **MEDIUM** / 미통과 시: 저장 항목 재수행.

미통과 항목이 있으면 실패 규격에 따라 기록 후 복귀 조건을 확인한다.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 추정·리스크에 기록한다.

### 이 STEP의 심각도 적용 예시

→ [SEVERITY_RULE.md](../../common/SEVERITY_RULE.md) 참조

- **CRITICAL**: 사실과 추정을 섞어 원인 정보 자체를 신뢰할 수 없다 → STEP2 또는 STEP3으로 복귀해 사실 관계 재확인
- **HIGH**: 사용자 결론, 점수, 근거 범위가 없어 보고서만 보고 다음 행동을 결정할 수 없다 → 보고서 재작성 후 진행
- **MEDIUM**: 다음 단계 제안 또는 우선순위 근거가 빈약하다 → 근거 보강 후 조건부 진행
- **LOW**: finding 심각도 근거 문장이 짧아 판단 근거가 약하다 → 진행 가능, 다음 보고서에서 보강

## 산출물

STEP 완료 시 아래 형식으로 값을 채워 출력한다.
이전 STEP 산출물 요약을 상단에 포함해 다음 워크플로우가 맥락 없이 시작하지 않도록 한다.

용어 기준:
- 핵심 파일: STEP1에서 분석 시작점으로 선정한 파일
- 근거 파일: STEP2 이후 실제로 열어 확인한 파일 또는 공식 URL
- 직접 파일: 분석 결론 또는 후속 조치의 직접 대상
- 간접 파일: 직접 파일 때문에 함께 영향을 받는 대상

```text
[분석 요약]
진입 경로 : STEP3 통과 / STEP3 생략 직행 중 하나 — 근거
타입      : 분석 타입 — 선정 이유
핵심 파일 : 파일명:라인번호 / 진입점 — 선정 이유
근거 파일 : 파일명:라인번호 또는 공식 URL — 읽은 이유
직접 파일 : 파일명:라인번호 — 역할
간접 파일 : 파일명:라인번호 — 영향 이유. 없으면 "없음"
영향 계층 : 계층명 — 파일명:라인번호 (STEP3에서 그대로)
기준 문서 : 파일명:라인번호 또는 공식 URL — 기준 역할. 없으면 "없음" (STEP3에서 그대로)
사용자 영향 : 사용자에게 보이는 영향. 없으면 "사용자 영향 없음 (확인 완료)" (STEP3에서 그대로)

[사용자 요약]
한 줄 결론 : 결론 텍스트
점수       : Agent 수행성 00/100, 사용자 이해성 00/100
감점 이유  : Agent 수행성 감점 이유 (없으면 "없음") / 사용자 이해성 감점 이유 (없으면 "없음")
범위       : 읽은 범위 / 제외 범위 / 미확인 범위
우선 행동  : 1순위 — 개선 또는 다음 행동
             2순위 — 개선 또는 다음 행동

[STEP4 보고]
사실      : 사실 목록 (근거 포함)
추정      : 추정/가정 목록 (없으면 "없음")
원인      : 1순위 — 원인 설명
             2순위 — 원인 설명
심각도    : finding명 — 등급 / 근거 / 다음 행동
다음 단계 : 주 워크플로우 (+ 보조 워크플로우) — 근거 설명
저장 경로 : docs/agents/output/analysis/YYYYMMDD_HHmmss.md
```

→ 출력 예시는 [STEP4_SAMPLES.md](STEP4_SAMPLES.md) 참조

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `사실`의 각 항목에 파일명:라인번호 또는 공식 문서 근거가 있는가 — 근거 없는 항목이 있으면 `추정`으로 이동해야 한다
- [ ] `감점 이유`가 `범위`의 미확인 범위, `리스크`와 연결되는가
- [ ] `다음 단계`의 주 워크플로우가 CRITICAL/HIGH finding의 워크플로우와 일치하는가 — 불일치하면 주/보조 판정 기준을 재확인한다
- [ ] `우선 행동`의 1순위가 `원인` 1순위 원인과 직접 연결되는가

## 복귀 조건

STEP4는 분석의 마지막 단계다. 이슈 원인에 따라 각기 다른 STEP으로 복귀한다.

- 사실/추정 분리가 불가능하다. 근거가 부족하다 → [STEP2_READ.md](../step2/STEP2_READ.md)로 복귀, `근거 파일` / `호출 흐름` 재정리
- 원인 우선순위를 결정할 수 없다. 영향 범위가 불명확하다 → [STEP3_IMPACT.md](../step3/STEP3_IMPACT.md)로 복귀, `영향 계층` / `직접 파일` 재정리
- STEP3 생략 직행 경로에서 직접 파일·간접 파일·사용자 영향이 불명확하다 → [STEP2_READ.md](../step2/STEP2_READ.md)로 복귀, `[STEP4 직행 보강]` 재작성
- 보고 중 핵심 대상 파일 누락을 확인했다 → [STEP1_SCOPE.md](../step1/STEP1_SCOPE.md)로 복귀, `핵심 파일` 재확정
- STEP1 타입 분류가 잘못됐음이 최종 보고 단계에서 확인됐다 → [STEP1_SCOPE.md](../step1/STEP1_SCOPE.md)로 복귀, 타입 재분류 후 STEP2부터 재수행
- 같은 원인으로 동일 STEP에 2회 이상 복귀한다 → 작업 중단, 사용자에게 실패 규격 전체 보고

## 다음 STEP

CRITICAL finding 발생 시:

1. 현재까지 완성된 산출물(`[분석 요약]` + `[사용자 요약]` + `[STEP4 보고]`)을 먼저 출력한다.
2. CRITICAL finding을 별도 블록으로 사용자에게 보고한다.
3. 사용자 응답을 받기 전까지 다음 워크플로우 진입을 보류한다.

하네스 통과와 산출물 출력이 완료되면 분석을 종료한다. `다음 단계`에 따라 다음 워크플로우로 이동한다.
