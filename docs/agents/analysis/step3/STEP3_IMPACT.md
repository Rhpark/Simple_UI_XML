# STEP3. 영향 범위

목적: 분석 타입에 맞춰 계층별 책임과 후속 조치 시 영향을 받을 파일, 모듈, 사용자 범위를 정리한다.

## 진입 조건

STEP2 하네스를 통과한 산출물이 전달됐을 때 진입한다.

아래 9개 항목이 모두 채워져 있어야 한다. 하나라도 비어 있으면 STEP2로 복귀한다.

- 타입 (보조 타입 포함)
- 핵심 파일 (주/보조 구분)
- 제외 범위/시나리오
- 근거 파일
- 분기 분석
- 호출 흐름 (code-flow 타입이면 필수, 아니면 "해당 없음")
- 문서 주장 (doc-consistency 타입이면 필수, 아니면 "해당 없음")
- 리스크
- 인계 항목

## 행동 원칙

- STEP2에서 확인된 근거만 영향 범위 판단의 기반으로 사용한다.
- 직접/간접 파일 분류는 파일명:라인번호 근거와 함께 기록한다.
- 추측으로 계층을 채우지 않는다. 근거가 없으면 미확인으로 기록한다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. STEP2에서 전달된 항목을 [입력 수신] 블록에 기록한다.
  → 타입·핵심 파일·근거 파일 모두 채워짐: 계속 진행한다.
  → 하나 이상 비어 있음: STEP2로 복귀해 누락 항목을 재확정한다.
- [ ] 2. 분석 타입에 맞는 읽기 근거를 바탕으로 영향 계층을 식별하고 근거와 함께 기록한다.
  → code-flow: `호출 흐름`의 각 구간이 어느 계층(presentation / domain / data)에 해당하는지 파일명:라인번호 근거와 함께 식별한다.
  → doc-consistency: `문서 주장`을 근거로 source of truth / dependent doc / backup doc / test doc 계층을 식별한다.
  → dependency-env: build config / dependency catalog / official source / local runtime 계층을 근거와 함께 식별한다.
  → ui-spec: screen / state / copy / guard / empty state 계층을 근거와 함께 식별한다.
  → product-scope: product scope / phase scope / policy / excluded scope 계층을 근거와 함께 식별한다.
  → 계층 식별 불가: 리스크에 CRITICAL로 기록하고 STEP2로 복귀해 `호출 흐름` 또는 `문서 주장`을 재추적한다.
- [ ] 3. 공통/기반 파일(util, base, 공유 유틸리티, DI 모듈 등) 포함 여부를 별도로 확인한다.
  → 포함됨: 공통 파일 목록에 파일명:라인번호와 역할을 기록한다.
  → 확인 완료 후 없음: "없음 (확인 완료)"으로 기록한다.
  → 확인 불가: 리스크에 HIGH로 기록하고 이유를 명시한다.
- [ ] 4. 직접 파일(분석 결론 또는 후속 조치의 직접 대상)과 간접 파일(후속 조치 시 파급 대상)을 분리해 기록한다.
       판단 기준: 직접 = 이번 분석 결론을 바꾸거나 후속 조치에서 직접 다룰 파일. 간접 = 직접 파일의 결론 또는 후속 조치로 동작·문서·검증 범위가 달라질 수 있는 파일(호출 대상, 테스트, 의존 문서 포함).
  → 같은 파일이 직접과 간접 두 목록에 들어가면: 직접으로만 기록한다.
  → 직접 파일이 0개: 리스크에 CRITICAL로 기록하고 STEP2로 복귀해 근거 파일을 재확인한다.
  → 간접 파일의 각 항목이 ①파일명:라인번호 근거 또는 ②'추론 — 근거 파일:라인번호 기반 이유' 중 하나인지 확인한다. 어느 쪽도 없으면 해당 항목을 제거하거나 리스크에 '미확인 간접 파일 — 이유'로 이동한다.
- [ ] 5. 영향을 받을 모듈, 테스트, 사용자 범위를 파악한다.
  → 테스트 파일 확인됨: 테스트 목록에 기록한다.
  → 비즈니스 로직 또는 상태 변경 파일에 테스트 파일이 없음: 리스크에 "테스트 없음 — 회귀 확인 불가"를 HIGH로 기록한다. 단순 위임 함수, 데이터 홀더, 자동 생성 파일은 이 기준에서 제외한다.
  → 테스트 없음 (확인 완료): "테스트 영향 없음 (확인 완료)"으로 기록한다.
- [ ] 6. 사용자에게 보이는 영향이 있는지 확인한다.
  → 있음: 사용자 영향에 구체적인 영향 내용을 기록한다.
  → 없음: "사용자 영향 없음 (확인 완료)"으로 기록한다.
- [ ] 7. 기준 문서 또는 공식 기준이 있는지 확인한다.
  → 있음: 기준 문서에 파일명:라인번호 또는 공식 URL과 기준 역할 설명을 기록한다.
  → 없음: "없음"으로 기록한다.
  → 있는데 미확인: 리스크에 "source of truth 미확인 — 이유"를 MEDIUM으로 기록한다.
- [ ] 8. doc-consistency 또는 dependency-env 타입이면 검증 범위를 확인한다. (해당 시)
  → 검증 범위 있음: 계속 진행한다.
  → 검증 범위 없음: 리스크에 실제 검증 필요성을 MEDIUM 이상으로 기록한다.
  → 해당 타입이 아님: 이 항목을 건너뛴다.
- [ ] 9. STEP2에서 전달된 리스크와 인계 항목을 이번 영향 범위 분석에서 처리 가능한지 확인한다.
  → 영향 범위 내에서 처리 가능: 해당 항목을 영향 범위와 연결해 리스크에 기록한다.
  → 처리 불가 (STEP4에서 다뤄야 함): 리스크에 흡수하고 "(STEP2 인계)"를 표시한다.
  → 리스크·인계 항목 모두 없음 (확인 완료): 계속 진행한다.
- [ ] 10. 미분석 입력 조합, 추가 확인 필요 호출부, 보류 리스크를 심각도와 함께 기록한다.
  → 없음: 계속 진행한다.
  → 있음: 리스크에 심각도와 함께 기록한다.
- [ ] 11. STEP4 진입 조건(타입, 근거 파일, 직접 파일, 간접 파일, 사용자 영향)이 산출물에 채워졌는지 최종 확인한다.
  → 모두 채워짐: STEP4로 진입한다.
  → 하나 이상 비어 있음: 해당 항목을 보완하거나 1번~10번 중 해당 항목으로 복귀한다.

## 하네스

- [ ] 1번 통과 — STEP2에서 전달된 타입·핵심 파일·근거 파일이 [입력 수신] 블록에 기록됐다.
     **CRITICAL** / 미통과 시: 1번 재수행. 채울 수 없으면 STEP2로 복귀해 누락된 항목 재확정.
- [ ] 2번 통과 — 타입에 맞는 영향 계층이 파일명:라인번호 근거와 함께 명시됐다.
     **CRITICAL** / 미통과 시: 계층 식별 불가면 STEP2로 복귀해 호출 흐름 또는 문서 주장 재추적.
- [ ] 3번 통과 — 공통/기반 파일 포함 여부가 확인됐다. 없으면 "없음 (확인 완료)"으로 명시됐다.
     **HIGH** / 미통과 시: 3번 재수행.
- [ ] 4번 통과 — 직접 파일이 1개 이상 기록됐고, 직접/간접 파일이 별도 목록으로 정리됐다. 간접 파일의 각 항목에 파일명:라인번호 근거 또는 '추론 — 이유'가 있다.
     **CRITICAL** / 미통과 시: 직접 파일 0개면 STEP2로 복귀해 근거 파일 재확인. 간접 파일 근거·추론 표시 누락이면 4번 재수행.
- [ ] 5번 통과 — 테스트 범위가 확인됐다. 비즈니스 로직 파일에 테스트가 없으면 리스크에 HIGH로 기록됐다.
     **HIGH** / 미통과 시: 5번 재수행.
- [ ] 6번 통과 — 사용자 영향 또는 "사용자 영향 없음 (확인 완료)"이 기록됐다.
     **MEDIUM** / 미통과 시: 6번 재수행.
- [ ] 7번 통과 — 기준 문서 또는 "없음/미확인"이 기록됐다.
     **MEDIUM** / 미통과 시: 7번 재수행.
- [ ] 8번 통과 — doc-consistency/dependency-env 타입이면 검증 범위가 확인됐다. 해당 없으면 자동 통과.
     **MEDIUM** / 미통과 시: 8번 재수행.
- [ ] 9번 통과 — STEP2 리스크·인계 항목이 처리됐거나 "없음 (확인 완료)"이다.
     **HIGH** / 미통과 시: 9번 재수행.
- [ ] 10번 통과 — 보류 리스크가 기록됐거나 없음이 확인됐다.
     **LOW** / 미통과 시: 10번 재수행.
- [ ] 11번 통과 — STEP4 진입 조건 5개(타입·근거 파일·직접 파일·간접 파일·사용자 영향)가 모두 채워졌다.
     **CRITICAL** / 미통과 시: 11번 재수행. 채울 수 없으면 누락 원인에 따라 해당 항목으로 복귀.

미통과 항목이 있으면 실패 규격에 따라 기록 후 복귀 조건을 확인한다. 루프 판정 기준은 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 참조

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 리스크에 기록한다.

### 이 STEP의 심각도 적용 예시

→ [SEVERITY_RULE.md](../../common/SEVERITY_RULE.md) 참조

- **CRITICAL**: 계층 식별 자체가 불가능해 직접 파일을 특정할 수 없다 → STEP2로 복귀해 근거 재추적
- **HIGH**: 공통/기반 파일(DI 모듈 등) 포함 여부를 확인하지 않은 채 영향 범위를 확정했다 → 확인 후 재작성
- **HIGH**: 기준 문서가 있는데 확인하지 않고 문서 충돌 결론을 냈다 → 기준 문서 확인 후 재작성
- **MEDIUM**: 직접/간접 파일은 분리했으나 테스트 범위를 빠뜨렸다 → 테스트 범위 추가 후 조건부 진행
- **LOW**: 보류 리스크 항목은 있지만 근거 문장이 약하다 → 진행 가능, 다음 보고서에서 보강

## 산출물

STEP 완료 시 아래 형식으로 값을 채워 출력한다.

용어 기준:
- 핵심 파일: STEP1에서 분석 시작점으로 선정한 파일
- 근거 파일: STEP2에서 실제로 열어 확인한 파일 또는 공식 URL
- 직접 파일: 분석 결론 또는 후속 조치의 직접 대상
- 간접 파일: 직접 파일 때문에 함께 영향을 받는 대상
- 공통 파일: 여러 기능이나 계층이 공유하는 기반 파일

```text
[입력 수신]
타입      : STEP2 "타입"에서 — 타입 / 선정 이유 (보조 타입이 있으면 "주: XX / 보조: YY")
핵심 파일 : STEP2 "핵심 파일"에서 — 파일명 / 선정 이유 (주/보조 구분)
근거 파일 : STEP2 "근거 파일"에서 — 파일명:라인번호

[STEP3 영향 범위]
영향 계층   : 계층명 — 호출 흐름 / 문서 주장 중 해당 구간 (파일명:라인번호)
               여러 계층이면 중요도 순으로 나열
기준 문서   : 파일명:라인번호 또는 공식 URL — 기준 역할 설명
               없으면 "없음", 미확인이면 "미확인 — 이유"
직접 파일   : 파일명:라인번호 — 역할 설명
               분석 결론 또는 후속 조치의 직접 대상. 여러 개면 중요도 순으로 나열
간접 파일   : 파일명:라인번호 — 영향 이유
               후속 조치 시 파급되는 파일/모듈/테스트. 없으면 "없음"
공통 파일   : 파일명:라인번호 — 공통/기반 파일 설명
               확인 완료 후 없으면 "없음 (확인 완료)"
사용자 영향 : 사용자에게 보이는 영향 설명
               없으면 "사용자 영향 없음 (확인 완료)"
테스트      : 테스트 파일/검증 항목 — 영향 이유
               없으면 "테스트 영향 없음 (확인 완료)"
리스크      : 심각도 — 리스크 설명. STEP2 인계 항목은 "(STEP2 인계)" 표시. 없으면 "없음"

[다음 STEP 진입 조건 충족 확인]
- [ ] 타입        — 채워짐 (STEP4 진입 조건)
- [ ] 근거 파일   — 채워짐 (STEP4 진입 조건)
- [ ] 직접 파일   — 채워짐 (STEP4 진입 조건)
- [ ] 간접 파일   — 채워짐 또는 "없음" (STEP4 진입 조건)
- [ ] 사용자 영향 — 채워짐 또는 "사용자 영향 없음" (STEP4 진입 조건)
```

               data — saveRecord(RecordRepository.kt:38) Room insert 호출

직접 파일   : CheckInViewModel.kt:142 — 저장 로직 진입점
               RecordRepository.kt:38 — Room insert 호출부
간접 파일   : RecordDao.kt:21 — Room 쿼리 변경 시 영향

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `직접 파일`에 있는 파일이 실제 이번 분석 결론 또는 후속 조치의 직접 대상인가 — 결론이 달라지지 않는 파일이 포함됐다면 간접으로 이동해야 한다
- [ ] `간접 파일`에 있는 파일이 `직접 파일`의 결론 또는 후속 조치로 실제로 동작·문서·검증 범위가 달라질 수 있는 파일인가 — 호출 관계나 의존 관계를 직접 확인했는가
- [ ] `리스크`의 HIGH 항목이 비즈니스 로직 또는 상태 변경 파일에만 적용됐는가 — 단순 위임 함수나 데이터 홀더에 HIGH가 붙었다면 기준 초과다
- [ ] `사용자 영향`이 실제 사용자 화면 또는 동작 변화로 이어지는 내용인가 — 내부 구조 변경인데 사용자 영향으로 기록됐다면 재확인이 필요하다

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- 영향 계층 식별이 불가능하고 STEP2로 복귀해도 근거를 확보할 수 없다 → 사용자에게 현재 상태·원인·선택지 보고
- 직접 파일이 0개이고 재추적 후에도 특정할 수 없다 → 사용자에게 분석 범위 재정의 요청
- 루프 판정 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용

## 복귀 조건

- 호출 체인 또는 근거 읽기가 불완전해 계층 책임을 식별할 수 없다 → [STEP2_READ.md](../step2/STEP2_READ.md)로 복귀, `호출 흐름 / 문서 주장` 재추적
- 공통/기반 파일 영향 확인 때문에 분석 범위가 달라진다 → [STEP1_SCOPE.md](../step1/STEP1_SCOPE.md)로 복귀, `대상 기능/모듈` 재확정
