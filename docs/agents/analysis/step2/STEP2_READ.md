# STEP2. 근거 읽기(흐름 추적)

목적: STEP1이 확정한 분석 타입에 맞는 근거를 직접 열어 확인하고, 코드 흐름, 문서 주장, 환경 조건을 STEP3·STEP4가 검증 가능한 형태로 정리한다.

## 진입 조건

STEP1 하네스를 통과한 산출물이 전달됐을 때 진입한다.

아래 9개 항목이 모두 채워져 있어야 한다. 하나라도 비어 있으면 STEP1로 복귀한다.

- 타입 (보조 타입 포함)
- 복원 매핑
- 대상 기능/모듈
- 핵심 파일 (주/보조 구분)
- 제외 범위/시나리오
- STEP3 적용 여부
- 입력 fixture/재현 조건
- 분석 환경
- 이슈

## 행동 원칙

- 직접 열어 확인한 코드, 문서, 실행 결과만 사실로 기록한다.
- 확인하지 않은 내용을 확인한 것처럼 서술하지 않는다.
- 호출 체인, 문서 주장, 버전 숫자는 파일명:라인번호 또는 공식 문서 URL 근거를 함께 기록한다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.
- 근거 유형(코드 / 주석 / 문서 / 로그 / 실행 결과)을 명시한다. code-flow 동작 설명은 실제 코드 근거를 우선한다.

## 실행 체크리스트

- [ ] 1. STEP1에서 전달된 9개 항목을 [입력 수신] 블록에 그대로 옮겨 기록한다.
  → 9개 항목 모두 채워짐: 계속 진행한다.
  → 하나 이상 비어 있음: STEP1으로 복귀해 누락 항목을 재확정한다.
- [ ] 2. 핵심 파일을 열어 진입점이 실제로 존재하는지 확인하고, 진입점 코드/문서를 근거 파일의 첫 항목으로 기록한다.
  → 진입점 존재: 파일명:라인번호를 근거 파일에 기록한다. 이후 Bash로 라인번호 실존을 검증한다.
     ```bash
     wc -l < "파일경로"
     ```
     반환값 ≥ 라인번호 → 실존 확인, 계속 진행한다.
     반환값 < 라인번호 → 파일을 다시 열어 올바른 라인번호로 수정 후 재기록한다.
  → 진입점 없음 (파일 자체가 없음): STEP1으로 복귀해 핵심 파일을 재확정한다.
- [ ] 3. STEP1 타입과 실제로 열어본 진입점 파일의 성격이 일치하는지 확인한다.
  → 일치: 계속 진행한다.
  → 불일치(예: code-flow 타입인데 진입점이 설정·문서 파일, doc-consistency 타입인데 진입점이 비즈니스 로직 코드): 이슈에 "STEP1 타입 불일치 의심 — 타입명 / 파일명"을 기록하고 사용자에게 확인을 요청한다. 확인 후 STEP1 복귀 여부를 결정한다.
- [ ] 4. 제외 범위/시나리오에 명시된 파일·모듈·시나리오를 추적 금지 경계로 표시한다.
       이후 5번~9번 추적 도중 경계에 닿으면 아래 분기를 적용한다.
  → 경계 안으로 들어가지 않아도 추적 완료 가능: 경계를 유지하고 계속 진행한다.
  → 경계 안으로 반드시 들어가야 추적 가능: 추적을 중단하고 STEP1으로 복귀해 제외 범위 또는 핵심 파일을 재확정한다.
- [ ] 5. 분석 타입에 맞는 읽기 방식을 선택해 적용한다.
  → 추적 도중 STEP1 타입 분류와 맞지 않는 강한 증거 발견 시: 추적을 중단하고 이슈에 "STEP1 타입 재검토 필요 — 발견 내용"을 기록한 뒤 STEP1 복귀를 판단한다.
  → 단일 타입: 해당 타입의 읽기 방식을 실행한다.
  → 주/보조 혼합 타입: 주 타입은 반드시 전체 읽기 방식을 실행한다. 보조 타입은 결론 보강에 필요한 최소 근거만 읽고, 읽지 않은 보조 범위는 리스크 또는 인계 항목에 이유와 함께 기록한다.
  → code-flow: 이벤트 → 함수 호출 → 상태 변경 → 렌더 반영 순서로 호출 체인을 직접 따라가며 호출 흐름을 기록한다. DI 인터페이스를 만났을 때:
	1. 인터페이스 선언부를 기록한다.
	2. DI 바인딩 모듈, 생성자 주입, provider, test fake를 검색한다.
	3. 구현체가 하나로 특정되면 구현체를 열어 추적한다.
	4. 구현체가 여러 개이면 fixture/환경 기준으로 선택한다.
	5. 그래도 특정 불가하면 HIGH 리스크로 기록하고 그 지점에서 중단한다.
  → doc-consistency: 기준 문서(source of truth), 의존 문서, 백업 문서를 분리해 읽는다.
  → dependency-env: 실제 설정 파일, 공식 문서, 현재 실행 결과를 분리해 읽고 버전/환경 조건을 기록한다.
  → ui-spec: 화면 상태, 사용자 행동, 문구, 차단/빈 상태를 기준 문서와 대조한다.
  → product-scope: 포함/제외 범위, 단계 구분, 정책 문구의 충돌을 확인한다.
- [ ] 6. 입력 fixture/재현 조건을 확인한다.
  → "없음": 이 항목을 건너뛰고 7번으로 진행한다.
  → 값 있음: 해당 입력값·재현 조건을 분기 추적의 입력 조건으로 사용한다. code-flow에서는 fixture가 통과하는 분기를 정상 분기로 잡고, dependency-env에서는 fixture가 적용된 실행 결과를 우선 근거로 사용한다.
- [ ] 7. 분석 환경을 확인한다.
  → 해당 OS·SDK·빌드 환경 값을 dependency-env 읽기 방식의 비교 기준으로 사용한다. 설정 파일 / 공식 문서 / 환경 값 세 축에서 충돌이 발견되면 충돌 항목에 기록한다.
- [ ] 8. 읽기 결과의 분기/차이 구조를 파악한다.
  → code-flow: 함수 진입점 하나당 정상 1건·예외 1건·경계 1건의 분기가 존재하는지 확인하고 각 분기의 입력 조건과 결과를 기록한다. 조건문이 없는 단순 위임 함수는 "경계·예외 해당 없음 — 위임 함수"로 기록한다.
  → doc-consistency·ui-spec·product-scope: 같은 기준에 대해 각 소스가 어떻게 다르게 주장하는지 일치/불일치 구조를 기록한다.
  → dependency-env: 설정 파일·공식 문서·실행 결과 세 축의 값이 일치하는지 확인하고 불일치 항목을 기록한다.
- [ ] 9. 아래 두 가지를 별도로 확인한다.
  - 예외·경계 처리 방식: 8번에서 확인된 예외·경계 분기가 어떻게 처리되는지 기록한다(null 반환 / 예외 throw / 상태 변경 통지 / 로그만 등). 비동기 실행 컨텍스트이면 취소 예외가 catch에 포함됐는지도 확인하고, 포함됐다면 rethrow 여부를 기록한다.
  - 횡단 관심사: 생명주기, 동시성, 비동기, source of truth, 환경 호환성 영향 지점을 확인한다.
  영향 없으면 각각 "확인 완료 — 영향 없음"으로 기록한다.
- [ ] 10. 5번~9번 추적 도중 구현체 또는 기준 문서를 특정할 수 없는 지점에 닿으면 아래 분기를 적용한다.
  → 구현체 특정 가능: 구현 클래스를 열어 추적을 계속한다.
  → 구현체 특정 불가 (DI 인터페이스는 5번의 5단계에서 처리됨): 리스크에 "구현체 미확인 — 파일명:라인번호"를 HIGH로기록하고 해당 지점에서 추적을 중단한다.
  → 기준 문서 특정 가능: 해당 문서를 열어 문서 주장에 기록한다.
  → 기준 문서 특정 불가: 리스크에 "source of truth 미확인 — 이유"를 MEDIUM으로 기록한다.
- [ ] 11. 문서 주장은 각 소스가 주장하는 기준을 있는 그대로 기록하고, 다른 주장과 모순되는 경우에만 충돌 항목에 기록하면서 해당 주장 항목 끝에 "(→ 충돌 기록됨)"을 표시한다. dependency-env 타입이면 근거 유형(공식 문서 / 로컬 실행)도 함께 명시한다. 읽기 도중 결정되지 않은 기준이나 미확정 값(TODO, TBD, 빈 필드 등)을 발견하면 미정 항목에 파일명:라인번호와 함께 기록한다. 없으면 "없음 (확인 완료)"으로 명시한다.
- [ ] 12. STEP1 이슈의 각 항목을 STEP2에서 해소했는지 확인한다. 읽기로 해소된 항목은 근거 파일에서 해소 근거를 가리키고, 해소 못 한 항목은 리스크에 흡수해 심각도와 함께 기록한다. STEP3 이후에서 다뤄야 할 항목은 인계 항목에 별도 인계한다.
  → 해소된 항목: 근거 파일에 해소 근거(파일명:라인번호)를 명시한다.
  → 해소 못 한 항목(STEP2 범위 내): 리스크에 흡수하고 심각도를 부여하며 "(이슈 흡수)"를 표시한다.
  → STEP3 이후에서 처리할 항목: 인계 항목에 항목 내용과 인계 사유를 함께 기록한다.
  → 이슈가 "없음": 리스크와 인계 항목을 "없음 (확인 완료)"으로 명시한다.
- [ ] 13. STEP3 진입 조건(분석 타입, 근거 파일, 분기 분석) 또는 STEP4 직행 진입 조건(분석 타입 + 선정 이유, 근거 파일, 직접 파일, 간접 파일, 사용자 영향)이 산출물에 채워졌는지 마지막으로 확인한다.
  → STEP3 적용 여부 = 필수: [다음 STEP 진입 조건 충족 확인] 블록에서 STEP3 관련 항목이 모두 채워졌는지 확인하고 STEP3으로 진입한다.
  → STEP3 적용 여부 = 생략: STEP3 진입 조건 확인을 건너뛴다. [STEP4 직행 보강] 블록의 직접 파일, 간접 파일, 사용자 영향이 채워졌는지 확인하고 STEP4로 직접 진입한다.

## 하네스

- [ ] 1번 통과 — STEP1에서 전달된 9개 항목(타입·복원 매핑·대상 기능/모듈·핵심 파일·제외 범위/시나리오·STEP3 적용 여부·입력 fixture/재현 조건·분석 환경·이슈)이 [입력 수신] 블록에 빠짐없이 기록됐다.
     누락 항목: [없음 또는 누락 항목명을 여기 기재 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 1번 재수행. 채울 수 없으면 STEP1로 복귀해 누락된 항목 재확정.
- [ ] 2번 통과 — 핵심 파일의 진입점 파일명과 라인번호가 실제로 열려 확인됐고 근거 파일에 기록됐다. Bash로 라인번호 실존이 검증됐다.
     확인된 진입점: [파일명:라인번호를 여기 기재 — 비어있으면 미통과]
     Bash 검증: [총 라인 수 N줄, 진입점 LN 실존 확인 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 2번 재수행. 파일이 존재하지 않거나 열 수 없으면 STEP1로 복귀해 핵심 파일 재확정.
- [ ] 3번 통과 — STEP1 타입과 진입점 파일의 성격이 확인됐다. 불일치가 발견됐으면 이슈에 기록됐고 사용자 확인이 요청됐다.
     **HIGH** / 미통과 시: 3번 재수행. 타입 불일치가 확인됐으면 STEP1으로 복귀해 타입 재확정.
- [ ] 4번 통과 — 제외 범위/시나리오가 추적 금지 경계로 표시됐고, 추적 결과가 그 경계를 침범하지 않았다.
     **HIGH** / 미통과 시: 4번 재수행. 경계 침범이 필수면 STEP1로 복귀해 제외 범위/시나리오 또는 핵심 파일 재확정.
- [ ] 5번 통과 — 분석 타입에 맞는 읽기 방식이 적용됐다(code-flow는 이벤트→상태→렌더, doc-consistency는 기준/의존/백업 분리, dependency-env는 설정/공식/실행 분리, ui-spec은 화면/행동/문구/차단 대조, product-scope는 범위/단계/정책 충돌 확인). 혼합 타입이면 주 타입 전체 읽기와 보조 타입 처리 방식(읽음/리스크/인계)이 기록됐다.
     **HIGH** / 미통과 시: 5번 재수행. 다른 타입 방식으로 결론을 냈다면 STEP1로 복귀해 분석 타입 재확정.
- [ ] 6번 통과 — 입력 fixture/재현 조건이 "없음(해당 없음/미확인)"이 아닐 때 해당 값이 분기 추적의 입력 조건으로 실제 사용됐고 분기 분석에 반영됐다. "없음(해당 없음)" 또는 "없음(미확인)"이면 해당 표시가 있다.
     **MEDIUM** / 미통과 시: 6번 재수행.
- [ ] 7번 통과 — 분석 환경이 "없음"이 아닐 때 해당 값이 설정/공식/실행 비교 축에 사용됐고 문서 주장 또는 충돌 항목에 반영됐다. "없음"이면 해당 표시가 있다.
     **MEDIUM** / 미통과 시: 7번 재수행.
- [ ] 8번 통과 — 함수 진입점 하나당 정상 1건·예외 1건·경계 1건이 분기 분석에 기록됐거나, 해당 없음 이유가 명시됐다. 각 분기 항목에 파일명:라인번호가 있거나 '해당 없음 — 이유'가 명시됐다.
     **HIGH** / 미통과 시: 8번 재수행. 단순 위임 함수면 "해당 없음 — 위임 함수"로 명시.
- [ ] 9번 통과 — 예외 경로 및 생명주기·동시성·비동기·source of truth·환경 호환성 영향 지점이 확인됐다. 영향 없으면 "확인 완료 — 영향 없음"으로 명시됐다.
     **MEDIUM** / 미통과 시: 9번 재수행.
- [ ] 10번 통과 — 구현체 미확인·기준 문서 미확인이 발생한 경우 리스크에 기록됐고 추적 중단 지점이 명시됐다. 발생하지 않았으면 "없음 (확인 완료)"이다.
     **HIGH** / 미통과 시: 10번 재수행.
- [ ] 11번 통과 — 문서 주장과 충돌 항목이 분리됐고, 충돌이 있는 주장 항목에 "(→ 충돌 기록됨)" 표시가 일치한다.
     **MEDIUM** / 미통과 시: 11번 재수행.
- [ ] 12번 통과 — 이슈의 각 항목이 (a) 근거 파일로 해소, (b) 리스크에 흡수, (c) 인계 항목으로 인계 중 하나로 처리됐다. 이슈가 "없음"이면 리스크와 인계 항목 모두 "없음 (확인 완료)" 또는 실제 항목이다.
     **HIGH** / 미통과 시: 12번 재수행.
- [ ] 13번 통과 — STEP3 적용 여부가 필수이면 STEP3 진입 조건(분석 타입, 근거 파일, 분기 분석)이 모두 채워졌다. STEP3 적용 여부가 생략이면 STEP4 직행 진입 조건(분석 타입 + 선정 이유, 근거 파일, 직접 파일, 간접 파일, 사용자 영향)이 모두 채워졌다.
     진행 경로: [STEP3 필수 / STEP4 직행 중 여기 기재 — 비어있으면 미통과]
     미충족 항목: [없음 또는 미충족 항목명을 여기 기재 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 13번 재수행. 채울 수 없으면 누락 원인에 따라 1~12번 중 해당 항목으로 복귀.

미통과 항목이 있으면 실패 규격에 따라 기록 후 강제 중단 조건 또는 복귀 조건을 확인한다.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 리스크/인계 항목에 기록한다.

### Bash 사용 규칙

허용:
- wc, sed, cat, head, tail
- rg, grep, find
- git diff, git status
- 빌드 도구 테스트·lint 실행은 사용자 허용 시에만

금지:
- 파일 수정 명령
- rm, mv, cp로 프로젝트 파일 변경
- 빌드 도구로 대규모 산출물 삭제
- dependency install/update
- formatter 자동 적용

## 산출물(STEP 3 또는 STEP 4에 전달 정보)

STEP 완료 시 아래 형식으로 값을 채워 출력한다.

용어 기준:
- 핵심 파일: STEP1에서 분석 시작점으로 선정한 파일
- 근거 파일: STEP2에서 실제로 열어 확인한 파일 또는 공식 URL
- 직접 파일: 분석 결론 또는 후속 조치의 직접 대상
- 간접 파일: 직접 파일 때문에 함께 영향을 받는 대상

```text
[입력 수신]
타입                   : STEP1 "타입"에서 — 타입 / 선정 이유 (보조 타입이 있으면 "주: XX / 보조: YY")
복원 매핑              : STEP1 "복원 매핑"에서 — 원문 표현 → 복원 결과. 없으면 "없음"
대상 기능/모듈         : STEP1 "대상 기능/모듈"에서 — 진입점
핵심 파일              : STEP1 "핵심 파일"에서 — 파일명 / 선정 이유 (주/보조 구분)
제외 범위/시나리오     : STEP1 "제외 범위/시나리오"에서 — 추적 금지 경계 (없으면 "없음")
STEP3 적용 여부        : STEP1 "STEP3 적용 여부"에서 — 필수 / 생략
입력 fixture/재현 조건 : STEP1 "입력 fixture/재현 조건"에서 (없으면 "없음(해당 없음)" 또는 "없음(미확인)")
분석 환경              : STEP1 "분석 환경"에서 (없으면 "없음")
이슈                   : STEP1 "이슈"에서 — 내용 또는 "없음" (처리 방식은 STEP2 체크리스트 12번에서 결정)

[STEP2 근거 읽기]
읽기 방식   : 분석 타입과 동일한 값
              혼합 타입이면 "주: XX 전체 읽기 / 보조: YY 최소 근거 확인 또는 인계" 형식으로 기록
근거 파일   : 파일명:라인번호 또는 공식 문서 URL — 읽은 이유 / 근거 유형(코드 / 주석 / 문서 / 실행 결과)
호출 흐름   : code-flow이면 이벤트명 → 함수명(파일명:라인번호) → 상태 변경 → 렌더 반영
              code-flow가 아니면 "해당 없음"
문서 주장   : 문서/설정/코드가 주장하는 기준 (파일명:라인번호) / 근거 유형
              다른 주장과 모순되면 항목 끝에 "(→ 충돌 기록됨)" 표시
분기 분석   : 정상 — 조건 설명 (파일명:라인번호)
              예외 — 조건 설명 (파일명:라인번호)
              경계 — 조건 설명 (파일명:라인번호) / 해당 없으면 "해당 없음 — 이유"
충돌 항목   : 문서A:라인 / 문서B:라인 — 충돌 내용. 없으면 "없음 (확인 완료)"
미정 항목   : 파일명:라인번호 — 비어 있는 결정 또는 미확정 기준. 없으면 "없음"
리스크      : 심각도 — 리스크 설명. 이슈에서 흡수된 항목은 "(이슈 흡수)" 표시
              확인됐으나 없으면 "없음 (확인 완료)", 확인 못 했으면 "미확인 — 이유"
인계 항목   : STEP3 이후에서 다뤄야 할 이슈 잔여 항목 — 인계 사유. 없으면 "없음 (확인 완료)"

[STEP4 직행 보강]  ※ STEP3 적용 여부 = 생략일 때만 작성
직접 파일   : 파일명:라인번호 — 분석 결론 또는 후속 조치의 직접 대상. 없으면 "없음 — 이유"
간접 파일   : 파일명:라인번호 — 직접 파일의 결론 또는 후속 조치로 함께 영향을 받는 파일/문서. 없으면 "없음 (확인 완료)"
사용자 영향 : 사용자에게 보이는 영향 설명. 없으면 "사용자 영향 없음 (확인 완료)"

[다음 STEP 진입 조건 충족 확인]
- [ ] 타입          — 채워짐 (STEP3·STEP4 진입 조건)
- [ ] 근거 파일     — 채워짐 (STEP3·STEP4 진입 조건)
- [ ] 분기 분석     — 채워짐 (STEP3 적용 여부 = 필수일 때 STEP3 진입 조건)
- [ ] 타입 선정 이유 — 포함됨 (STEP4 진입 조건)
- [ ] STEP4 직행 보강 — 채워짐 (STEP3 적용 여부 = 생략일 때 STEP4 진입 조건)
```

→ 출력 예시는 [STEP2_SAMPLES.md](STEP2_SAMPLES.md) 참조

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다. 하네스가 다루지 않는 외부 검증 항목이다.

- [ ] 근거 파일의 파일명:라인번호를 실제 파일에서 열었을 때 근거 코드/문장이 해당 줄에 있는가.
- [ ] 호출 흐름의 각 화살표(→)가 서로 다른 함수 또는 파일을 넘어가는가 — 같은 파일 내 두 줄이 화살표로 연결됐다면 계층 경계가 잘못 표시된 것이다.

## 실패 규격

하네스 미통과 시 아래 형식으로 기록한다.

```text
실패 STEP   : STEP2
실패 항목   : 미통과 하네스 항목명(번호 포함)
실패 원인   : 어떤 데이터가 없는지 / 어떤 조건이 충족되지 않는지
복귀 대상   : 하네스 항목에 명시된 복귀 체크리스트 번호 또는 STEP1 복귀 사유
전달 데이터 : 완성된 [입력 수신] + [STEP2 근거 읽기] 필드 값. 미완성 항목은 "미완성"
```

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- 대상 파일이 존재하지 않거나 열 수 없어 근거 읽기 자체가 불가능하다 → 사용자에게 핵심 파일 재확인 요청
- STEP1 산출물에서 채울 수 있는 값이 절반 이상 비어 있다 → 사용자에게 STEP1 재실행 요청
- 루프 판정 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용
