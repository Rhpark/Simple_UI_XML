# STEP5. 최종 리뷰 보고

목적: STEP1~4에서 수집한 findings를 중복 없이 병합해 심각도 순으로 정리하고, 리뷰 하네스를 통과하지 못한 finding은 해당 STEP으로 복귀해 재검토한다.

## 진입 조건

- STEP4 하네스를 통과했다
- STEP4 산출물 전체(`[분석 입력]` + `[기능 검증]` + `[로직 & 안정성]` + `[아키텍처]` + `[코드 품질]`)가 전달됐다
  - [ ] `[기능 검증] 발견 사항` — STEP1 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[로직 & 안정성] 발견 사항` — STEP2 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[아키텍처] 발견 사항` — STEP3 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[코드 품질] 발견 사항` — STEP4 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[로직 & 안정성] 확인 범위`, `[아키텍처] 확인 범위`, `[코드 품질] 확인 범위` — 각 단계의 확인/제외/미확인 범위가 기록돼 있다
  - [ ] `[코드 품질] 도구 실행` — 도구 실행 결과 또는 미실행 이유가 기록돼 있다
  - [ ] `[기능 검증] 호환성` — 호환성 확인 결과 또는 "해당 없음"이 기록돼 있다
  - [ ] `[코드 품질] 테스트 코드` — 테스트 코드 확인 결과가 기록돼 있다
  - [ ] `[코드 품질] 개발자 편의성` — 개발자 편의성 확인 결과가 기록돼 있다

## 행동 원칙

- STEP1~4 산출물에 기록된 finding만 병합 대상으로 삼는다. 이 단계에서 새 finding을 만들지 않는다.
- 파일명:라인번호, 코드 근거, 심각도 판정 근거, 사용자 영향이 빠진 finding은 출력하지 않는다. 누락 항목은 원본 STEP으로 복귀해 재확인한다.
- CRITICAL finding은 사용자 응답 없이 다음 워크플로우로 진행하지 않는다.
- 중복 제거 시 가장 직접적인 STEP을 대표로 두고 나머지는 보조 근거로 표시한다. 같은 원인인지 판단이 어려우면 별개 finding으로 유지한다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. 보고서 첫 줄에 `현재 단계: 리뷰 - STEP5 최종 리뷰 보고`를 명시한다.
- [ ] 2. 최종 보고서 상단에 한 줄 결론(`[리뷰 요약] 한 줄 결론`)을 쓴다.
- [ ] 3. `Agent 수행성`과 `사용자 이해성`을 각각 100점 기준으로 평가해 `[리뷰 요약] 점수`에 기록한다.
- [ ] 4. `[리뷰 요약] 감점 이유`에 감점 이유를 findings, 확인 범위, 도구 실행 결과와 연결해 기록한다.
- [ ] 5. STEP1~4의 확인 범위를 병합해 읽은 범위, 제외 범위, 미확인 범위를 `[리뷰 요약] 확인 범위`에 기록한다.
- [ ] 6. `[리뷰 요약] 우선 행동`에는 점수와 무관하게 병합된 findings 중 사용자가 먼저 처리해야 할 우선 조치를 최대 3개까지 기록한다.
  → finding이 없거나 모두 `무시 가능`이면 "없음"으로 기록한다.
- [ ] 7. STEP1~4의 모든 findings를 한 곳에 모아 중복 여부를 확인한다.
  → 같은 파일명:라인번호에 여러 STEP의 finding이 겹침: 가장 직접적인 STEP 1건만 남기고 나머지는 보조 근거로 표시한다.
  → 중복 없음: 모든 finding을 그대로 유지한다.
- [ ] 8. 병합된 finding 목록을 아래 기준으로 정렬한다.
  - 1순위: CRITICAL → HIGH → MEDIUM → LOW 순
  - 같은 등급 내에서는 사용자 영향이 큰 순
- [ ] 9. 각 finding이 아래 리뷰 하네스를 통과하는지 확인한다.
  - 파일명:라인번호가 기록됐다
  - 코드 근거 (실제 코드 줄 기반)가 기록됐다
  - 심각도 판정 근거가 기록됐다
  - 사용자 영향이 기록됐다
- [ ] 10. 리뷰 하네스를 통과하지 못한 finding이 있으면 해당 STEP으로 복귀해 재확인한다.
- [ ] 11. CRITICAL 등급 finding이 있으면 현재까지 완성된 산출물을 먼저 출력하고 CRITICAL finding을 별도 블록으로 사용자에게 보고한다.
- [ ] 12. 각 finding에 다음 행동을 기록한다.
  → `feature`: 외부 동작 변경이 필요한 경우.
  → `refactor`: 외부 동작 유지하면서 구조 개선이 필요한 경우.
  → `무시 가능`: 사용자 영향이 없고 LOW 등급인 경우에만 허용.
- [ ] 13. 산출물 전체를 `docs/agents/output/review/YYYYMMDD_HHmmss.md` 형식으로 저장한다. STEP6 실행 시 동일 파일을 덮어쓴다. (날짜·시각은 현재 기준으로 채운다)

## 하네스 (통과 기준)

- [ ] 1번 통과 — STEP1~4의 모든 finding이 중복 없이 병합됐다.
     병합된 finding 총수: [숫자를 여기 기재 — 비어있으면 미통과 / 없으면 "0건"]
     **HIGH** / 미통과 시: 1번 재수행. STEP1~4 `발견 사항`을 순서대로 대조해 누락·중복 항목 재확인.
- [ ] 2번 통과 — finding이 CRITICAL → HIGH → MEDIUM → LOW 순으로 정렬됐다.
     **MEDIUM** / 미통과 시: 2번 재수행. `[리뷰 보고] 발견 사항` 목록을 심각도 순으로 재정렬.
- [ ] 3번 통과 — 모든 finding에 파일명:라인번호 + 코드 근거 + 심각도 근거 + 사용자 영향이 기록됐다.
     누락 finding: [없음 또는 누락 finding명을 여기 기재 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 3번 재수행. 파일명:라인번호 없는 finding은 해당 finding의 원본 STEP으로 복귀해 코드 근거 재확인.
- [ ] 4번 통과 — 모든 finding에 다음 행동(feature / refactor / 무시 가능)이 기록됐다.
     **HIGH** / 미통과 시: 4번 재수행. 행동 미기록 finding에 feature / refactor / 무시 가능 중 하나를 판정 근거와 함께 기록.
- [ ] 5번 통과 — `[리뷰 요약] 한 줄 결론`, `[리뷰 요약] 점수`, `[리뷰 요약] 감점 이유`, `[리뷰 요약] 확인 범위`, `[리뷰 요약] 우선 행동`이 기록됐다.
     **HIGH** / 미통과 시: 5번 재수행. 누락된 `[리뷰 요약]` 항목을 STEP1~4 산출물 기반으로 재작성.
- [ ] 6번 통과 — CRITICAL finding이 있으면 산출물 출력 후 별도 블록으로 보고됐다.
     CRITICAL finding 존재 여부: [있음/없음을 여기 기재 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 6번 재수행. CRITICAL finding을 별도 블록으로 추출해 사용자에게 보고하고 응답 대기.
- [ ] 7번 통과 — 산출물이 docs/agents/output/review/ 에 저장됐다.
     저장 경로: [경로를 여기 기재 — 비어있으면 미통과]
     **MEDIUM** / 미통과 시: 13번 재수행. 파일 저장 후 경로 기록.

미통과 항목이 있으면 → 실패 규격에 따라 기록 후 해당 STEP으로 복귀.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 finding에 기록한다.

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `[리뷰 보고] 발견 사항`의 각 finding에서 파일명:라인번호를 직접 열었을 때 기록된 근거가 해당 줄에 실제로 있는가
- [ ] `[리뷰 요약] 감점 이유`의 감점 이유가 findings, 확인 범위, 도구 실행 결과와 연결되는가
- [ ] `[리뷰 요약] 확인 범위`의 미확인 범위가 최종 결론을 흔들 가능성은 없는가
- [ ] `[리뷰 보고] 중복 제거`에서 제거된 finding이 실제로 같은 원인인지, 별개 문제인지 다시 확인이 필요한가
- [ ] `[리뷰 보고] 다음 단계`의 워크플로우 방향이 analysis STEP4 `다음 단계`와 일치하는가 — 불일치하면 analysis 결과와 리뷰 결과 중 어느 쪽이 더 근거가 있는지 판단한다

## 실패 규격

하네스 미통과 시 아래 형식으로 기록한다.

```text
실패 STEP   : STEP5
실패 항목   : 미통과 하네스 항목명(번호 포함)
실패 원인   : 어떤 데이터가 없는지 / 어떤 조건이 충족되지 않는지
복귀 대상   : 하네스 N번 참조 또는 STEP1/STEP2/STEP3/STEP4 중 해당 finding의 원본 STEP
전달 데이터 : 완성된 [리뷰 요약] + [리뷰 보고] 섹션 값 (미완성 항목은 "미완성")
```

## 복귀 조건

- finding에 파일명:라인번호가 없다 → 해당 finding의 원본 STEP으로 복귀, 코드 근거 재확인
- 중복 제거 기준을 정할 수 없다 (대표 STEP 불명확) → 해당 finding이 발생한 두 STEP 중 상위 STEP으로 복귀
- 같은 finding으로 동일 STEP에 2회 이상 복귀한다 (루프 판정) → 작업 중단, 사용자에게 실패 규격 전체 보고

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- STEP4 산출물이 누락되거나 진입 조건 6개 항목이 절반 이상 비어 있다 → 사용자에게 STEP4 재실행을 요청한다.
- CRITICAL finding이 발견되어 별도 블록으로 보고한 뒤 사용자 응답을 받지 못했다 → 사용자 응답 대기, 다음 워크플로우 진입 보류.
- finding 위치를 끝까지 특정할 수 없다 (원본 STEP에서 확인 불가) → 사용자에게 finding 위치 재확인을 요청한다.
- 같은 finding으로 동일 STEP에 2회 이상 복귀하는 루프가 발생했다 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용 후 작업 중단.

## 다음 STEP

CRITICAL 등급 finding 발생 시:

1. 현재까지 완성된 산출물(`[리뷰 요약]` + `[리뷰 보고]`)을 먼저 출력한다
2. CRITICAL finding을 별도 블록으로 사용자에게 보고한다
3. 사용자 응답을 받기 전까지 다음 워크플로우 진입을 보류한다

하네스 통과 + 산출물 출력 완료 →
- 점수 평가 요청이 있는 경우: [STEP6_SCORE.md](../step6/STEP6_SCORE.md)로 진입한다.
- 요청이 없는 경우: 리뷰 종료. `[리뷰 보고] 다음 단계`에 따라 다음 워크플로우로 이동한다.
