# STEP1. 기능 검증

목적: 분석 결과를 입력으로 받아 기능 누락·과잉 구현·요청 의도 불일치를 판정한다.

전제: review 워크플로우는 반드시 analysis 워크플로우 완료 후 실행한다. analysis STEP4 하네스를 통과하지 못했거나 STEP4 `[STEP4 보고]` · `[사용자 요약]` · `[분석 요약]` 산출물이 없으면 review를 시작하지 않는다.

## 진입 조건

- analysis STEP4 하네스를 통과했다
- 아래 항목이 실제 값으로 채워진 상태로 전달됐다

  **analysis STEP4 `[분석 요약]`에서:**
  - [ ] `진입 경로` — STEP3 통과 또는 STEP3 생략 직행 여부가 기록돼 있다
  - [ ] `타입` — 분석 타입과 선정 이유가 기록돼 있다
  - [ ] `핵심 파일` — 파일명:라인번호와 선정 이유가 기록돼 있다
  - [ ] `근거 파일` — 파일명:라인번호 또는 공식 URL로 기록돼 있다
  - [ ] `직접 파일` — 파일명:라인번호로 기록돼 있다
  - [ ] `간접 파일` — 파일명:라인번호로 기록돼 있다. 없으면 "없음"으로 확인돼 있다
  - [ ] `영향 계층` — 기록돼 있다
  - [ ] `기준 문서` — 기록돼 있다. 없으면 "없음", 미확인이면 "미확인 — 이유"로 확인돼 있다
  - [ ] `사용자 영향` — 기록돼 있다. 없으면 "사용자 영향 없음 (확인 완료)"으로 확인돼 있다

  **analysis STEP4 `[사용자 요약]`에서:**
  - [ ] `한 줄 결론` — 기록돼 있다
  - [ ] `점수` — Agent 수행성 / 사용자 이해성 점수가 기록돼 있다
  - [ ] `감점 이유` — 각 점수의 감점 이유 또는 "없음"이 기록돼 있다
  - [ ] `범위` — 읽은 범위 / 제외 범위 / 미확인 범위가 기록돼 있다
  - [ ] `우선 행동` — 기록돼 있다. 없으면 "없음"으로 확인돼 있다

  **analysis STEP4 `[STEP4 보고]`에서:**
  - [ ] `사실` — 파일명:라인번호 근거와 함께 기록돼 있다
  - [ ] `추정` — 기록돼 있다. 없으면 "없음"으로 확인돼 있다
  - [ ] `원인` — 우선순위와 함께 기록돼 있다
  - [ ] `심각도` — finding별 등급 / 근거 / 다음 행동이 기록돼 있다
  - [ ] `다음 단계` — review / feature / refactor 중 하나로 기록돼 있다

## 행동 원칙

- 코드를 직접 열어 확인한 결과만 finding으로 기록한다.
- 파일명:라인번호 없는 finding은 기록하지 않는다.
- analysis 산출물에 없는 사실을 추정으로 채우지 않는다. 부족하면 analysis 단계로 복귀하거나 사용자에게 확인을 요청한다.
- 기능 충족 판정은 기준 문서 또는 사용자 요청에 근거해야 하며, 근거 없는 결론은 도출하지 않는다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. 보고서 첫 줄에 `현재 단계: 리뷰 - STEP1 기능 검증`을 명시한다.
- [ ] 2. analysis STEP4 `원인`의 1순위부터 순서대로 기능 충족 여부를 판정한다.
  → 1순위 원인이 기능 영역 밖(런타임/구조/컨벤션): STEP1에서는 보류하고 해당 STEP 소관으로 표시한다.
  → 기능 영역: 계속 판정한다.
- [ ] 3. analysis STEP4 `범위`를 먼저 확인해 읽지 않은 범위 또는 미확인 범위가 기능 판정에 영향을 주는지 기록한다.
  → 영향 있음: `[기능 검증] 검증 맥락`의 미확인 범위에 명시하고 판정 보류 범위를 표시한다.
  → 영향 없음: 계속 진행한다.
- [ ] 4. analysis STEP4 `[분석 요약] 기준 문서`가 있으면 기능 기대 동작을 해당 기준 문서와 대조한다.
  → 기준 문서 없음/미확인: `[기능 검증] 검증 맥락` 기준 문서에 "없음/미확인"으로 기록하고 사용자 요청 자체를 기준으로 사용한다.
- [ ] 5. 사용자가 기대한 동작이 실제 코드에 구현됐는지 analysis STEP4 `직접 파일`을 직접 열어 확인한다.
  → 직접 파일이 존재하지 않거나 열 수 없음: 복귀 조건 적용.
- [ ] 6. analysis STEP4 `[분석 요약] 사용자 영향`을 기준으로 finding의 사용자 영향을 기록한다.
- [ ] 7. 아래 엣지 케이스 목록을 기능 관점에서 하나씩 확인한다 (해당 없음 판정: 해당 입력 타입 자체가 존재하지 않거나 도달 불가능한 경로일 때만 허용)
  - null / empty 입력 (null을 받을 수 있는 파라미터·필드가 있는가)
  - 0개 데이터 (리스트·컬렉션 반환이 있는가)
  - 단일 항목 (리스트·컬렉션 처리가 있는가)
  - 최대 개수 (상한 제한이 코드에 명시됐는가 — 없으면 "상한 없음" 기록)
  - 실패 응답 (네트워크 오류, DB 오류, 타임아웃)
- [ ] 8. 입력값 유효성 검사가 누락됐는지, 누락이 사용자 기대 결과를 깨뜨리는지 판정한다.
- [ ] 9. 요청보다 많은 기능이 구현됐는지 (과잉 구현) 확인한다.
- [ ] 10. 환경 호환성을 확인한다.
  → [REVIEW_STEP1_CHECKLIST.md](../../platforms/android/review/step1/REVIEW_STEP1_CHECKLIST.md) <!-- ★ --> 
- [ ] 11. 발견한 문제를 1차 판정 우선순위 기준에 따라 분류한다.
  → 사용자 기대 결과를 깨뜨림: STEP1 소관 finding으로 기록한다.
  → 런타임 오류·안정성 문제: STEP2 소관으로 표시하고 STEP1에서는 기록하지 않는다.
  → 구조·컨벤션 문제: STEP3/STEP4 소관으로 표시하고 STEP1에서는 기록하지 않는다.

## 하네스 (통과 기준)

- [ ] 1번 통과 — analysis 산출물(`[사용자 요약] 한 줄 결론`, `[STEP4 보고] 원인`, `[사용자 요약] 범위`, `[분석 요약] 직접 파일`, `[분석 요약] 사용자 영향`, `[분석 요약] 기준 문서`, `[분석 요약] 영향 계층`)을 입력으로 사용했다.
     사용된 analysis 항목: [실제 인용한 analysis 항목명을 여기 기재 — 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 1번 재수행. analysis STEP4 산출물 없이 진행 불가 → review 중단 후 analysis 먼저 완료.
- [ ] 2번 통과 — 기능 판정에 사용한 기준 문서 또는 "기준 문서 없음/미확인"이 기록됐다.
     **HIGH** / 미통과 시: 2번 재수행. `[분석 입력] 기준 문서` 항목을 기준으로 기준 문서 유무 재확인.
- [ ] 3번 통과 — 분석 미확인 범위가 기능 판정에 영향을 주는지 확인했다.
     **HIGH** / 미통과 시: 3번 재수행. `[기능 검증] 검증 맥락` 확인 범위를 기준으로 미확인 범위 영향 재판정.
- [ ] 4번 통과 — 엣지 케이스 5종이 모두 확인됐다 (확인 완료 후 해당 없으면 "해당 없음 — 이유" 명시).
     **HIGH** / 미통과 시: 4번 재수행. null/empty·0개·단일 항목·최대 개수·실패 응답 중 미확인 항목을 코드에서 직접 확인.
- [ ] 5번 통과 — 발견된 finding마다 파일명:라인번호, 근거, 심각도, 사용자 영향이 기록됐다.
     확인된 finding 위치: [파일명:라인번호를 여기 기재 — finding이 "없음"이면 "없음", 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 5번 재수행. 파일명:라인번호 없는 finding은 코드에서 직접 확인 후 위치 기록.
- [ ] 6번 통과 — 1차 판정 우선순위 규칙에 따라 STEP2/STEP3/STEP4 소관 문제는 이 단계에서 기록하지 않았다.
     **MEDIUM** / 미통과 시: 6번 재수행. STEP1 finding 목록에서 런타임·구조·컨벤션 문제를 이관 항목으로 이동.
- [ ] 7번 통과 — 호환성 확인 결과가 기록됐다.
     호환성 확인 결과: [감지된 API 목록 또는 "해당 없음 (API 미사용)"을 여기 기재 — 비어있으면 미통과]
     **MEDIUM** / 미통과 시: 7번 재수행. 직접 파일에서 플랫폼 API 사용 여부 재확인 후 결과 기록.
     → [REVIEW_STEP1_HARNESS.md](../../platforms/android/review/step1/REVIEW_STEP1_HARNESS.md) <!-- ★ -->

미통과 항목이 있으면 → 실패 규격에 따라 기록 후 복귀 조건 확인.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 finding에 기록한다.

## 산출물

STEP 완료 시 아래 형식으로 값을 채워 출력한다.
이 산출물 전체가 다음 STEP으로 그대로 전달된다.

```text
[분석 입력]
직접 파일    : (analysis STEP4 `[분석 요약] 직접 파일` 그대로 — 파일명:라인번호 목록)
영향 계층    : (analysis STEP4 `[분석 요약] 영향 계층` 그대로 — 계층 정보)
기준 문서    : (analysis STEP4 `[분석 요약] 기준 문서` 그대로 — 파일명:라인번호 또는 "없음/미확인")

[기능 검증]
검증 대상  : 검증한 기능명 또는 시나리오명 (analysis `[분석 요약] 핵심 파일`의 진입점 기준)
검증 맥락  : 분석 요약 — analysis `한 줄 결론`
             기준 문서 — [분석 입력] 기준 문서 또는 "없음/미확인"
             확인 범위 — 읽은 범위 / 제외 범위 / 미확인 범위
엣지 케이스: null/empty — 확인 결과
             0개 데이터 — 확인 결과
             단일 항목 — 확인 결과
             최대 개수 — 확인 결과
             실패 응답 — 확인 결과
             (해당 없으면 "해당 없음 — 이유")
발견 사항  : finding명 — 등급 / 파일명:라인번호 / 근거 / 사용자 영향
             (finding이 여러 개면 줄 단위로 나열, 없으면 "없음")
호환성    : 1줄 요약 또는 "해당 없음 (API 미사용)"
            → [REVIEW_STEP1_OUTPUT.md](../../platforms/android/review/step1/REVIEW_STEP1_OUTPUT.md) <!-- ★ -->
이관 항목  : STEP2 소관 — 항목 설명 (없으면 "없음")
             STEP3 소관 — 항목 설명
             STEP4 소관 — 항목 설명
```

→ 추가 예시 및 실패 규격 예시는 [STEP1_SAMPLES.md](STEP1_SAMPLES.md) 참조

예시:

```text
[분석 입력]
직접 파일    : CheckInViewModel.kt:142 / RecordRepository.kt:38
영향 계층    : presentation(ViewModel) → data(Repository) → local(RoomDB)
기준 문서    : RECORD_LOGIC_SPEC.md:2

[기능 검증]
검증 대상  : 감정 기록 저장 기능
검증 맥락  : 분석 요약 — 저장 실패 시 UI 상태가 갱신되지 않을 가능성이 있다
             기준 문서 — RECORD_LOGIC_SPEC.md:2 저장 가능 조건 기준
             확인 범위 — 읽은 범위: ViewModel/Repository / 제외 범위: Calendar / 미확인 범위: 없음
엣지 케이스: null/empty — emotionTag null 시 저장 차단 확인 (CheckInViewModel.kt:148)
             0개 데이터 — causeTags 빈 리스트 허용, 저장 정상 완료 확인
             단일 항목 — intensity=1 최솟값 저장 정상 동작
             최대 개수 — 해당 없음 — causeTags 상한 제한 없음
             실패 응답 — DB 오류 시 UI 무응답 (롤백 없음) — HIGH
발견 사항  : 저장 실패 무응답 — HIGH / CheckInViewModel.kt:155 / catch 없이 emit 없음 / 사용자가 저장 실패를 인지 불가
호환성    : 해당 없음 (플랫폼 API 미사용)
이관 항목  : STEP2 소관 — viewModelScope 취소 시 insert 중단 가능성 (RecordRepository.kt:38)
             STEP3 소관 — 없음
             STEP4 소관 — 없음
```

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `[기능 검증] 엣지 케이스`에서 "해당 없음"으로 기록된 항목의 이유가 실제 코드 구조와 일치하는가
- [ ] `[기능 검증] 검증 맥락`의 기준 문서와 확인 범위가 analysis 산출물과 일치하는가
- [ ] `[기능 검증] 발견 사항`의 파일명:라인번호를 직접 열었을 때 기록된 근거가 해당 줄에 실제로 있는가
- [ ] `[기능 검증] 이관 항목`의 STEP2 소관 항목이 실제로 런타임 문제이지 기능 문제가 아닌가
- [ ] `[기능 검증] 호환성`에서 "해당 없음"으로 기록된 경우 실제로 플랫폼 API를 사용하지 않는가. 보강 OUTPUT 블록의 검증 포인트도 함께 확인했는가.
       → [REVIEW_STEP1_OUTPUT.md](../../platforms/android/review/step1/REVIEW_STEP1_OUTPUT.md) <!-- ★ -->

## 실패 규격

하네스 미통과 시 아래 형식으로 기록한다.

```text
실패 STEP   : STEP1
실패 항목   : 미통과 하네스 항목명(번호 포함)
실패 원인   : 어떤 데이터가 없는지 / 어떤 조건이 충족되지 않는지
복귀 대상   : 하네스 N번 참조 또는 analysis STEP4 복귀 사유
전달 데이터 : 완성된 [기능 검증] 섹션 값 (미완성 항목은 "미완성")
```

## 복귀 조건

- analysis 산출물이 불완전해 기능 판정 기준을 세울 수 없다 → analysis STEP4로 복귀, `사실` / `원인` / `범위` / `직접 파일` 재확인
- 검증 중 새로운 핵심 파일이 발견된다 → analysis [STEP1_SCOPE.md](../../analysis/step1/STEP1_SCOPE.md)로 복귀, `scope.target` 재확정

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- analysis STEP4 산출물 자체가 없거나 진입 조건 12개 항목이 절반 이상 비어 있다 → 사용자에게 review 보류와 analysis 재실행을 안내한다.
- 직접 파일이 존재하지 않거나 열 수 없어 기능 검증 자체가 불가능하다 → 사용자에게 핵심 파일 재확인을 요청한다.
- CRITICAL finding이 발견됐고 사용자 응답 없이 다음 STEP으로 진행해야 하는 상황이다 → 사용자 응답 대기.
- 루프 판정 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용.
