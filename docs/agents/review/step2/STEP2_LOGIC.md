# STEP2. 로직 & 안정성

목적: 기능이 존재해도 로직 오류·예외 처리 문제·런타임 안정성 문제로 동작이 깨지는지 판정한다.

## 진입 조건

- STEP1 하네스를 통과했다
- STEP1 산출물 전체(`[분석 입력]` + `[기능 검증]`)가 전달됐다
  - [ ] `[기능 검증] 발견 사항` — STEP1 기능 검증 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[기능 검증] 검증 맥락` — 분석 요약, 기준 문서, 확인 범위가 기록돼 있다
  - [ ] `[기능 검증] 이관 항목` — STEP2 소관으로 넘긴 항목이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[분석 입력] 직접 파일` — 직접 관련 파일 목록이 파일명:라인번호로 기록돼 있다

## 행동 원칙

- 코드를 직접 열어 확인한 결과만 finding으로 기록한다.
- 파일명:라인번호 없는 finding은 기록하지 않는다.
- 런타임 동작은 실제 코드 흐름에 근거해 판단한다. 추측으로 "오류가 발생할 것"이라고 기록하지 않는다.
- STEP1 이관 항목 외에도 직접 파일에서 새로 발견한 런타임 문제는 finding에 기록한다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. 보고서 첫 줄에 `현재 단계: 리뷰 - STEP2 로직 & 안정성`을 명시한다.
- [ ] 2. `[기능 검증] 이관 항목`의 STEP2 소관 항목을 먼저 확인한다.
  → 이관 항목 없음: 직접 파일 전수 검토로 진행한다.
  → 이관 항목 있음: 해당 항목을 우선 검증한 뒤 직접 파일 전수 검토로 확장한다.
- [ ] 3. `[기능 검증] 검증 맥락`의 기준 문서와 확인 범위를 확인해 런타임 판정에서 제외할 범위와 미확인 범위를 분리한다.
- [ ] 4. 아래 로직 오류 항목을 `[분석 입력] 직접 파일`에서 직접 확인한다 (확인 완료 후 없으면 "없음 (확인 완료)").
  - 잘못된 조건 분기 (반대 조건, 누락된 분기, 도달 불가능한 else)
  - 무한 루프 가능성 (종료 조건 없는 while, 재귀 호출)
  - null 처리 누락 (null 반환값을 null 안전 처리 없이 사용)
  - 경계값 누락 (off-by-one, 빈 컬렉션의 첫/마지막 원소 접근)
- [ ] 5. 아래 런타임 안정성 항목을 확인한다 (확인 완료 후 없으면 "없음 (확인 완료)").
  - 메모리 누수 (장기 보유 객체가 단기 생명주기 객체를 참조하는지)
  - 비동기 작업 누수 (라이프사이클과 무관한 스코프에서 시작된 작업, 취소되지 않는 작업)
  - 화면 단위 컴포넌트 소멸 이후 상태 변경
  - 예외를 삼키는 처리 (빈 catch 블록 `catch(e) {}`, 예외를 로그만 남기고 상위로 전파 안 함)
  → [REVIEW_STEP2_CHECKLIST.md](../../platforms/android/review/step2/REVIEW_STEP2_CHECKLIST.md) <!-- ★ -->
- [ ] 6. 발견한 문제를 1차 판정 우선순위 기준에 따라 분류한다.
  → 런타임 안정성·로직 오류: STEP2 소관 finding으로 기록한다.
  → 사용자 기대 결과를 깨뜨림: STEP1 소관으로 표시하고 STEP2에서는 기록하지 않는다.
  → 구조·컨벤션 문제: STEP3/STEP4 소관으로 표시하고 STEP2에서는 기록하지 않는다.

## 하네스 (통과 기준)

- [ ] 1번 통과 — `[기능 검증] 이관 항목`의 STEP2 소관 항목이 모두 확인됐다.
     **HIGH** / 미통과 시: 1번 재수행. STEP1 이관 항목을 `[로직 & 안정성]` 확인 항목과 대조해 미처리 항목 재확인.
- [ ] 2번 통과 — `[기능 검증] 검증 맥락`의 확인 범위를 반영해 미확인 런타임 범위가 기록됐다.
     **HIGH** / 미통과 시: 2번 재수행. `[로직 & 안정성] 확인 범위`에 미확인 런타임 범위 항목 추가.
- [ ] 3번 통과 — 로직 오류 4종 (조건 분기·무한 루프·null 처리·경계값)이 모두 확인됐다.
     **HIGH** / 미통과 시: 3번 재수행. 미확인 항목을 `[분석 입력] 직접 파일`에서 코드 레벨로 직접 확인.
- [ ] 4번 통과 — 런타임 안정성 4종 (메모리·비동기·생명주기·예외 처리)이 모두 확인됐다.
     **HIGH** / 미통과 시: 4번 재수행. 미확인 안정성 항목을 `[분석 입력] 직접 파일`에서 코드 레벨로 직접 확인.
     → [REVIEW_STEP2_HARNESS.md](../../platforms/android/review/step2/REVIEW_STEP2_HARNESS.md) <!-- ★ -->
- [ ] 5번 통과 — 발견된 finding마다 파일명:라인번호, 근거, 심각도, 사용자 영향이 기록됐다.
     확인된 finding 위치: [파일명:라인번호를 여기 기재 — finding이 "없음"이면 "없음", 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 5번 재수행. 파일명:라인번호 없는 finding은 코드에서 직접 확인 후 위치 기록.
- [ ] 6번 통과 — 1차 판정 우선순위 규칙에 따라 STEP1/STEP3/STEP4 소관 문제는 이 단계에서 기록하지 않았다.
     **MEDIUM** / 미통과 시: 6번 재수행. STEP2 finding 목록에서 기능·구조·컨벤션 문제를 이관 항목으로 이동.

미통과 항목이 있으면 → 실패 규격에 따라 기록 후 복귀 조건 확인.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 finding에 기록한다.

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `[로직 & 안정성] 로직 확인`에서 "없음 (확인 완료)"으로 기록된 항목의 이유가 실제 코드 구조와 일치하는가
- [ ] `[로직 & 안정성] 확인 범위`의 미확인 런타임 범위가 로직/안정성 결론을 흔들지 않는가
- [ ] `[로직 & 안정성] 발견 사항`의 파일명:라인번호를 직접 열었을 때 빈 catch 블록·생명주기 위반이 해당 줄에 실제로 있는가

## 복귀 조건

- STEP1 `[기능 검증] 이관 항목`이 불완전해 STEP2 소관 항목을 특정할 수 없다 → [STEP1_FUNC.md](../step1/STEP1_FUNC.md)로 복귀, `[기능 검증] 이관 항목` 재확인
- 로직 추적 중 범위 외 파일이 반드시 필요해진다 → analysis [STEP1_SCOPE.md](../../analysis/step1/STEP1_SCOPE.md)로 복귀, `scope.target` 범위 재확정

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- STEP1 산출물이 누락되거나 진입 조건 4개 항목이 절반 이상 비어 있다 → 사용자에게 STEP1 재실행을 요청한다.
- 직접 파일이 존재하지 않거나 열 수 없어 런타임 판정 자체가 불가능하다 → 사용자에게 핵심 파일 재확인을 요청한다.
- CRITICAL finding이 발견됐고 사용자 응답 없이 다음 STEP으로 진행해야 하는 상황이다 → 사용자 응답 대기.
- 루프 판정 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용.
