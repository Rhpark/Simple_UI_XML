# STEP6. 점수 평가

목적: STEP1~5 리뷰 결과를 바탕으로 6개 관점별 점수를 산출하고 코드 전반의 품질 수준을 정량화한다. 사용자가 "평가해줘", "점수화해줘" 등을 명시한 경우에만 실행하는 선택 단계다.

## 진입 조건

- STEP5 하네스를 통과했다
- 사용자가 점수 평가를 명시적으로 요청했다 ("평가해줘", "점수화해줘" 또는 유사 표현)
- STEP5 산출물 전체(`[리뷰 요약]` + `[리뷰 보고]`)가 전달됐다
  - [ ] `[리뷰 보고] 발견 사항` — finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[리뷰 요약] 검증 대상` — 검토 대상 파일·진입점이 기록돼 있다
  - [ ] `[리뷰 요약] 확인 범위` — 확인/제외/미확인 범위가 기록돼 있다
  - [ ] `[기능 검증] 호환성` — 호환성 확인 결과 또는 "해당 없음"이 기록돼 있다
  - [ ] `[코드 품질] 테스트 코드` — 테스트 코드 확인 결과가 기록돼 있다
  - [ ] `[코드 품질] 개발자 편의성` — 개발자 편의성 확인 결과가 기록돼 있다

## 행동 원칙

- STEP1~5 산출물에 기록된 finding만 감점 근거로 사용한다. 이 단계에서 새 finding을 만들지 않는다.
- 감점 계산은 감점 기준표를 기계적으로 적용하고, 정성 보정은 관점당 ±3점을 초과하지 않는다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 감점 기준표

관점별 finding 심각도에 따라 아래 점수를 감점한다. 최저 0점 (음수 불가).

| 심각도 | 기능(20) | 로직(20) | 아키텍처(20) | 품질(15) | 테스트(10) | 편의성(15) |
|---|---|---|---|---|---|---|
| CRITICAL | -8 | -8 | -8 | -6 | -4 | -6 |
| HIGH | -4 | -4 | -4 | -3 | -2 | -3 |
| MEDIUM | -2 | -2 | -2 | -1 | -1 | -1 |
| LOW | -1 | -1 | -1 | -1 | -1 | -1 |

정성 보정: 관점당 ±3점 상한. 적용 시 사유 명시 필수.

## 실행 체크리스트

- [ ] 1. 보고서 첫 줄에 `현재 단계: 리뷰 - STEP6 점수 평가`를 명시한다.
- [ ] 2. STEP5 산출물에서 finding과 확인 결과를 6개 관점별로 분류한다.
  - 기능 관점: `[기능 검증]` 소관 finding + `[기능 검증] 호환성` finding
  - 로직 관점: `[로직 & 안정성]` 소관 finding
  - 아키텍처 관점: `[아키텍처]` 소관 finding
  - 품질 관점: `[코드 품질]` 소관 finding (품질·컨벤션 finding, 테스트·편의성 제외)
  - 테스트 관점: `[코드 품질] 테스트 코드` finding
  - 편의성 관점: `[코드 품질] 개발자 편의성` finding
  → finding이 여러 관점에 걸치는 경우: STEP5 대표 STEP 기준으로 하나의 관점에만 배정한다.
- [ ] 3. 각 관점의 감점 기준표를 적용해 관점별 점수를 계산한다.
  - finding별 감점을 합산하고, 최저 0점 하한을 적용한다.
- [ ] 4. 정성 보정을 적용한다.
  - 관점당 ±3점 상한. 보정 적용 시 사유를 한 줄로 명시한다.
  → 보정 없음: "없음"으로 기록한다.
- [ ] 5. 관점별 점수를 합산해 종합 점수(0~100)를 산출하고 등급을 판정한다.
  - 90~100: S — 즉시 배포 가능
  - 75~89: A — 경미한 보완 후 배포 가능
  - 60~74: B — 개선 후 재검토 권고
  - 40~59: C — 주요 개선 필요
  - 39 이하: D — 전면 재검토 필요
- [ ] 6. 각 관점의 정성 보정 아래에 개선 항목을 위험도 순으로 최대 3개 기록한다.
  → CRITICAL finding이 있거나 만점 대비 70% 미만인 관점: 반드시 1개 이상 기록한다.
  → finding이 없는 관점: "없음"으로 기록한다.
- [ ] 7. STEP5 산출물의 `저장 경로` 파일을 현재 산출물 전체로 덮어쓴다.

## 하네스 (통과 기준)

- [ ] 1번 통과 — 6개 관점 모두 finding 분류 및 감점 계산 근거가 기록됐다.
     미기록 관점: [없음 또는 미기록 관점명을 여기 기재 — 비어있으면 미통과]
     **HIGH** / 미통과 시: 1번 재수행. STEP5 산출물에서 누락 관점의 finding과 확인 결과를 재분류.
- [ ] 2번 통과 — 정성 보정 사유가 관점별로 기록됐다 (보정 없으면 "없음").
     **MEDIUM** / 미통과 시: 2번 재수행. 보정 적용 관점에 사유 추가 기록.
- [ ] 3번 통과 — 종합 점수(0~100)와 등급(S/A/B/C/D)이 기록됐다.
     종합 점수: [점수를 여기 기재 — 비어있으면 미통과]
     **HIGH** / 미통과 시: 3번 재수행. 관점별 점수 합산 후 등급 재판정.
- [ ] 4번 통과 — CRITICAL finding이 있거나 만점 대비 70% 미만인 관점의 개선 항목이 기록됐다.
     필수 기록 관점: [없음 또는 미기록 관점명 — 비어있으면 미통과]
     **HIGH** / 미통과 시: 6번 재수행. 해당 관점의 개선 항목을 추가 기록한다.
- [ ] 5번 통과 — STEP5 저장 파일이 STEP6 산출물로 덮어씌워졌다.
     저장 경로: [경로를 여기 기재 — 비어있으면 미통과]
     **MEDIUM** / 미통과 시: 7번 재수행.

미통과 항목이 있으면 → 실패 규격에 따라 기록 후 복귀 조건 확인.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 finding에 기록한다.

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `[점수 평가] 관점별 점수`의 감점 내역이 STEP5 산출물의 각 관점 finding 수·등급과 일치하는가
- [ ] `[점수 평가] 정성 보정`의 사유가 코드 실제 상태에 근거한 것인가 (관점당 ±3점 상한 준수 여부 포함)
- [ ] `[점수 평가] 종합 점수`가 관점별 점수 합산값과 일치하는가
- [ ] 각 관점의 `개선 항목`에서 CRITICAL finding이 있거나 만점 대비 70% 미만인 관점이 누락 없이 기록됐는가

## 복귀 조건

- STEP5 finding 목록이 불완전해 관점별 분류가 불가능하다 → [STEP5_REPORT.md](../step5/STEP5_REPORT.md)로 복귀, `[리뷰 보고] 발견 사항` 재확인

## 강제 중단 조건

복귀 없이 즉시 작업을 멈춘다.

- STEP5 산출물이 누락됐거나 진입 조건 항목이 절반 이상 비어 있다 → 사용자에게 STEP5 재실행을 요청한다.
- 정성 보정이 관점당 ±3점을 초과하는 상황이다 → 사용자에게 보정 범위 초과를 보고하고 확인을 요청한다.
- 동일 원인으로 동일 STEP에 2회 이상 복귀했다 → [STEP_ROLLBACK.md](../../common/STEP_ROLLBACK.md) 루프 판정 기준 적용.
