# STEP3. 아키텍처

목적: 레이어 책임 분리 위반·모듈 경계 위반·설계 원칙 위반을 코드 기반으로 식별하고 기능 문제와 구조 문제를 분리해 기록한다.

## 진입 조건

- STEP2 하네스를 통과했다
- STEP2 산출물 전체(`[분석 입력]` + `[기능 검증]` + `[로직 & 안정성]`)가 전달됐다
  - [ ] `[기능 검증] 발견 사항` — STEP1 기능 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[기능 검증] 이관 항목` — STEP2 이후 소관 항목이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[로직 & 안정성] 발견 사항` — STEP2 로직 finding 목록이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[로직 & 안정성] 확인 범위` — STEP2 확인 범위와 미확인 런타임 범위가 기록돼 있다
  - [ ] `[로직 & 안정성] 이관 항목` — STEP3 소관으로 넘긴 항목이 기록돼 있다 (없으면 "없음" 확인)
  - [ ] `[분석 입력] 영향 계층` — analysis에서 확인된 계층 정보가 기록돼 있다
  - [ ] `[분석 입력] 기준 문서` — 아키텍처 또는 기준 문서가 기록돼 있다. 없으면 "없음" 또는 "미확인"으로 확인돼 있다

## 행동 원칙

- 코드를 직접 열어 확인한 결과만 finding으로 기록한다.
- 파일명:라인번호 없는 finding은 기록하지 않는다.
- 레이어 위반·모듈 경계 위반은 실제 import 경로나 참조 관계로 입증한다. "위반일 수 있다" 같은 추측은 기록하지 않는다.
- 기준 문서가 없으면 일반 원칙을 따르되, 프로젝트 고유 규칙을 추측해 위반으로 단정하지 않는다.
- 모호하거나 불확실한 항목은 추측으로 채우지 않고 사용자에게 질문한다.

## 실행 체크리스트

- [ ] 1. 보고서 첫 줄에 `현재 단계: 리뷰 - STEP3 아키텍처`를 명시한다.
- [ ] 2. `[로직 & 안정성] 이관 항목`의 STEP3 소관 항목을 먼저 확인한다.
  → 이관 항목 없음: 직접 파일 전수 검토로 진행한다.
  → 이관 항목 있음: 해당 항목을 우선 검증한 뒤 직접 파일 전수 검토로 확장한다.
- [ ] 3. `[분석 입력] 기준 문서`에서 아키텍처 기준 문서가 있는지 확인한다.
  → 있음: 해당 문서를 `[아키텍처] 기준 문서`에 기록한다.
  → 없음/미확인: "아키텍처 기준 문서 없음/미확인"을 `[아키텍처] 기준 문서`에 기록한다.
- [ ] 4. `[분석 입력] 영향 계층`을 기준으로 각 계층의 책임 분리를 아래 항목으로 확인한다.
  - presentation이 data 계층을 직접 참조하는지 (레이어 건너뜀)
  - 상위 계층이 하위 계층의 데이터 접근 객체를 직접 호출하는지
  - data 계층이 UI 상태를 직접 변경하는지
- [ ] 5. 모듈 경계 위반을 확인한다.
  - 접근 범위를 제한해야 할 심볼이 외부에 공개됐는지
  - 다른 모듈의 구현체를 직접 참조하는지 (인터페이스 대신)
- [ ] 6. 가변 상태 외부 노출을 확인한다.
  - 가변 상태 객체가 외부에 직접 노출됐는지
  - 불변 래퍼 패턴이 지켜졌는지
- [ ] 7. 단일 책임 원칙(SRP) 위반을 확인한다.
  - 하나의 클래스/함수가 여러 계층 책임을 동시에 처리하는지
- [ ] 8. 추상화 의존 원칙(DIP) 위반을 확인한다.
  - 구현체를 직접 의존하고 인터페이스를 사용하지 않는 곳이 있는지
  - DI 컨테이너를 통한 주입은 위반으로 보지 않는다 — 생성자·필드에서 구현체를 new/직접 참조하는 경우만 해당
- [ ] 9. 개방-폐쇄 원칙(OCP) 위반을 확인한다. (조건부 — 상속·추상 클래스·봉인 타입 구조가 있는 경우만)
  → 해당 구조 없음: "OCP — 해당 없음 (상속/추상화 구조 없음)"으로 기록하고 건너뛴다.
  → 해당 구조 있음: 새 타입 추가 시 기존 조건 분기를 수정해야 하는지 확인한다. 수정이 필요하면 위반.
- [ ] 10. 리스코프 치환 원칙(LSP) 위반을 확인한다. (조건부 — 상속 가능 클래스·abstract class·interface 구현체가 있는 경우만)
  → 해당 구조 없음: "LSP — 해당 없음 (상속 계층 없음)"으로 기록하고 건너뛴다.
  → 해당 구조 있음: 오버라이드 메서드가 기반 타입의 계약(예외 종류·반환 범위·사전조건)을 위반하는지 확인한다. 다른 예외를 던지거나 반환 범위를 좁히면 위반.
- [ ] 11. 인터페이스 분리 원칙(ISP) 위반을 확인한다. (조건부 — 인터페이스가 정의된 경우만)
  → 해당 구조 없음: "ISP — 해당 없음 (인터페이스 없음)"으로 기록하고 건너뛴다.
  → 해당 구조 있음: 구현체가 사용하지 않는 메서드를 인터페이스가 강제하는지 확인한다. 빈 구현(구현 예외 발생 / 빈 body)이 있으면 위반.
- [ ] 12. Android 아키텍처 추가 확인을 한다.
  → [REVIEW_STEP3_CHECKLIST.md](../../platforms/android/review/step3/REVIEW_STEP3_CHECKLIST.md) <!-- ★ -->
- [ ] 13. 발견한 문제를 1차 판정 우선순위 기준에 따라 분류한다.
  → 구조·설계 원칙 위반: STEP3 소관 finding으로 기록한다.
  → 사용자 기대 결과를 깨뜨림: STEP1 소관으로 표시하고 STEP3에서는 기록하지 않는다.
  → 런타임 안정성·로직 오류: STEP2 소관으로 표시하고 STEP3에서는 기록하지 않는다.
  → 컨벤션·네이밍 문제: STEP4 소관으로 표시하고 STEP3에서는 기록하지 않는다.

## 하네스 (통과 기준)

- [ ] 1번 통과 — `[로직 & 안정성] 이관 항목`의 STEP3 소관 항목이 모두 확인됐다.
     **HIGH** / 미통과 시: 1번 재수행. STEP2 이관 항목을 `[아키텍처]` 확인 항목과 대조해 미처리 항목 재확인.
- [ ] 2번 통과 — `[아키텍처] 기준 문서`에 기준 문서 또는 "없음/미확인"이 기록됐다.
     **HIGH** / 미통과 시: 2번 재수행. `[분석 입력] 기준 문서`를 기준으로 아키텍처 문서 유무 재확인 후 기록.
- [ ] 3번 통과 — 레이어 책임 분리 3종 (건너뜀·데이터 접근 객체 직접 호출·UI 직접 변경)이 모두 확인됐다.
     **HIGH** / 미통과 시: 3번 재수행. 미확인 항목을 `[분석 입력] 영향 계층` 기준으로 해당 파일에서 직접 확인.
- [ ] 4번 통과 — 모듈 경계·가변 상태 노출·SRP·DIP·OCP·LSP·ISP(해당 시)가 모두 확인됐다.
     SRP 판정: 하나의 클래스/함수가 2개 이상의 계층 책임을 동시에 처리하면 위반.
     DIP 판정: 생성자·필드에서 구현체를 new 또는 직접 참조하면 위반 (DI 컨테이너 주입 제외). 플랫폼별 DI 컨테이너 식별은 해당 플랫폼 보강 체크리스트를 따른다.
     OCP 판정: 새 타입 추가 시 기존 조건 분기 수정이 필요하면 위반. 상속·추상화 구조 없으면 "해당 없음".
     LSP 판정: 오버라이드 메서드가 기반 타입 계약과 다른 예외를 던지거나 반환 범위를 좁히면 위반. 상속 계층 없으면 "해당 없음".
     ISP 판정: 구현체가 사용하지 않는 메서드를 인터페이스가 강제하면 위반 (빈 구현 포함). 인터페이스 없으면 "해당 없음".
     **HIGH** / 미통과 시: 4번 재수행. 미확인 항목을 해당 파일에서 직접 확인 후 판정 근거 기록.
- [ ] 5번 통과 — 발견된 finding마다 파일명:라인번호, 근거, 심각도, 사용자 영향이 기록됐다.
     확인된 finding 위치: [파일명:라인번호를 여기 기재 — finding이 "없음"이면 "없음", 비어있으면 미통과]
     **CRITICAL** / 미통과 시: 5번 재수행. 파일명:라인번호 없는 finding은 코드에서 직접 확인 후 위치 기록.
- [ ] 6번 통과 — 1차 판정 우선순위 규칙에 따라 STEP1/STEP2/STEP4 소관 문제는 이 단계에서 기록하지 않았다.
     **MEDIUM** / 미통과 시: 6번 재수행. STEP3 finding 목록에서 기능·런타임·컨벤션 문제를 이관 항목으로 이동.
- [ ] 7번 통과 — Android 아키텍처 추가 확인(12번)이 완료됐다.
     **HIGH** / 미통과 시: 12번 재수행. REVIEW_STEP3_CHECKLIST.md 항목별 결과 재확인.
     → [REVIEW_STEP3_HARNESS.md](../../platforms/android/review/step3/REVIEW_STEP3_HARNESS.md) <!-- ★ -->
- [ ] 8번 통과 — 보고서 첫 줄에 `현재 단계: 리뷰 - STEP3 아키텍처` 헤더가 있다.
     **LOW** / 미통과 시: 체크리스트 1번 재수행.

미통과 항목이 있으면 → 실패 규격에 따라 기록 후 복귀 조건 확인.

우선순위 규칙: 체크리스트는 최소 보장선이지 상한선이 아니다. 결론을 바꿀 수 있는 사실을 발견하면 체크리스트 외 항목이라도 finding에 기록한다.

## 사용자 검증 포인트

AI 산출물 수령 후 아래 항목을 확인한다.

- [ ] `[아키텍처] 레이어 확인`에서 "없음 (확인 완료)"으로 기록된 항목이 실제 import 경로로 입증됐는가
- [ ] `[아키텍처] 기준 문서`가 실제 프로젝트 아키텍처 기준 문서와 일치하는가
- [ ] `[아키텍처] 확인 범위`의 제외/미확인 범위가 리뷰 결론을 흔들지 않는가
- [ ] `[아키텍처] 경계 확인`의 가변 상태 노출 항목에서 불변 래퍼 패턴이 실제 코드에서 지켜졌는가
- [ ] `[아키텍처] 발견 사항`의 SRP 위반 항목에서 해당 클래스가 실제로 2개 이상의 계층 책임을 처리하는지 import 경로로 확인할 수 있는가
- [ ] `[아키텍처] 발견 사항`의 DIP 위반 항목에서 DI 컨테이너 주입이 아닌 직접 참조(new / 직접 생성)인지 확인했는가
