# Severity Rule

이 파일은 analysis·review 두 워크플로우가 공유하는 심각도 4단계의 단일 정의 기준이다.
심각도 공통 정의는 이 파일에만 있다.

## SSOT 선언

심각도 4단계 공통 정의와 루프 판정 기준은 이 파일(`SEVERITY_RULE.md`)에만 존재한다.

---

## 4단계 정의

### CRITICAL

**판정 기준**
- 작업 결과 자체를 신뢰할 수 없다 — 사실·추정 혼재, 파일명:라인번호 근거 없음
- 핵심 기능이 동작하지 않거나 데이터 손실로 이어진다
- 계층·범위 식별이 불가능해 다음 STEP의 전제가 무너진다
- 존재하지 않는 API·파일·함수를 실제 호출·참조하고 있다 (hallucination)
- 병합 후에도 finding에 파일명:라인번호가 없다

**필수 행동**
- 즉시 작업 중단
- 해당 STEP 또는 이전 STEP으로 복귀해 근거 재확인
- 사용자에게 발견 내용·현재 상태·선택지를 보고한다
- 다음 STEP 진입 전 CRITICAL 항목이 해소돼야 한다

---

### HIGH

**판정 기준**
- 사용자에게 직접 노출되는 동작 오류가 있다
- 숨겨진 안정성 위협이 있다 — 비동기 누수, 생명주기 위반, 예외 삼킴(빈 catch 블록, 취소 예외를 포함하는 catch 등)
- 기준 문서가 있는데 확인하지 않고 결론을 도출했다
- 미확인 범위가 분석·리뷰 결론을 뒤집을 수 있다
- 가변 상태가 외부에 노출돼 예측 불가능한 상태 변경이 가능하다
- 엣지 케이스(null/empty/실패 응답)에서 사용자 기대 결과가 깨진다

**필수 행동**
- 해당 항목을 보완한 뒤 다음 STEP 진행
- 보완 불가 시 사용자에게 미보완 사유와 함께 보고 후 진행 여부 확인

---

### MEDIUM

**판정 기준**
- 기능은 동작하나 경계 처리·유효성 검사가 누락됐다
- 구조 문제가 있으나 현재 동작에는 영향이 없다 — SRP 위반, 중복 로직 등
- 다음 단계 제안·우선순위 근거가 빈약하다
- 컨벤션 문서가 없는데 컨벤션 위반 결론을 냈다
- source of truth가 있는데 확인하지 못했다

**필수 행동**
- 조건부 진행 가능
- 해당 항목을 리스크로 기록하고 계속 진행

---

### LOW

**판정 기준**
- 사용자 동작에 영향 없는 코드 품질 문제 — 미사용 import, 주석 코드, 네이밍 규칙 위반
- 근거 문장이 약하나 결론을 바꾸지 않는다
- 과잉 구현이 확인됐으나 사용자 동작에 영향 없다

**필수 행동**
- 진행 가능
- 해당 항목을 기록하고 다음 보고서에서 보강

---

## 심각도 판정 원칙

- finding에 심각도를 부여할 때 반드시 판정 근거(파일명:라인번호 또는 공식 URL)를 함께 기록한다.
- 같은 파일:라인번호에 여러 finding이 겹치면 가장 높은 심각도를 대표 등급으로 사용한다.
- 심각도는 현재 상태를 기준으로 판정한다. 미래 위험 가능성만으로 CRITICAL을 부여하지 않는다.
