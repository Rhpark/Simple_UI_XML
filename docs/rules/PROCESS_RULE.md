# PROCESS RULES

## 사용자 승인 기반 수정 원칙
 - 코드/문서 수정은 사용자의 명시적 승인 문구 확인 후에만 진행한다.
 - 사용자 명시 승인 전에는 읽기/분석 작업만 허용하며 파일 수정 명령을 실행하지 않는다.
 - 승인받은 범위를 벗어난 수정은 금지한다.

## 규칙 위반 감지 시 즉시 중단 절차
 - 규칙 위반 징후를 감지하면 즉시 모든 수정 작업을 중단한다.
 - 현재 상태(이미 변경된 파일, 미완료 작업, 예상 영향)를 사용자에게 즉시 보고한다.
 - 사용자 재승인 없이는 작업을 재개하지 않는다.

## 파일/패키지 생성 및 수정 규칙
 - 새 파일/패키지 생성 전, 반드시 기존 구조와 위치를 먼저 확인한다.
 - 패키지나 클래스의 실제 존재 여부를 Glob/Grep 도구로 검증한다.
 - 추정이나 가정으로 파일을 생성하지 않는다.
 - 불확실한 경우 반드시 사용자에게 확인 후 진행한다.

## 기능 추가 수정 시 검토 사항
 - 기능 추가/수정 전, 반드시 사이드 이펙트 발생 유무를 확인한다.
 - 기능 추가/수정 전, 수정 부분에 대해 반드시 사용자에게 고지할 것.
 - 기능 추가/수정 시, Android OS(SDK Version) 별 분기를 나눠야 할 필요가 있는지 반드시 확인한다.
 - 기능 추가/수정 시, 기존 작성된 코드 스타일과 비슷한 구조로 개발한다.


## 작업 전 성공 기준 정의
 - 작업 시작 전, 완료 판단 기준을 반드시 명시한다.
 - 다단계 작업은 아래 형식으로 계획을 제시하고 사용자 승인 후 진행한다.
   ```
   1. [작업 단계] → verify: [검증 방법]
   2. [작업 단계] → verify: [검증 방법]
   ```
 - 이 프로젝트의 표준 verify 기준:
   - API 변경 여부: `./gradlew apiCheck`
   - 코드 스타일: `./gradlew ktlintCheck`
   - 동작 검증: `./gradlew testRobolectric` 또는 `testUnit`
   - 빌드 통과: `./gradlew assemble`
 - "동작할 것 같다"는 추정으로 완료 선언하지 않는다.

## 문서 갱신 기준 (AGENT/RULE)

### AGENT/RULE 문서 갱신이 필요한 경우
 - 새로운 모듈 추가
 - 주요 아키텍처 변경 (예: 새로운 디자인 패턴 도입)
 - 개발 환경 변경 (Kotlin, Gradle 버전 업)
 - 새로운 개발 규칙 추가

### AGENT/RULE 문서 갱신이 불필요한 경우
 - 단순 버그 수정
 - 기존 기능 개선 (규칙 변경 없음)


## 파일/폴더 삭제 규칙
 - 파일/폴더 삭제 시 삭제 이유를 자세히 서술하고 사용자의 승인을 반드시 구한다.


## 대량 수정 작업 시 검증 프로세스
 - 대량 파일 수정 작업 전, 전체 범위와 수정 대상을 정확히 파악한다.
 - 각 단계별로 완전히 검증 후 다음 단계로 진행한다.
 - 최종 결과를 여러 방법(grep, 카운팅 등)으로 교차 검증한다.
 - 사용자가 직접 확인할 수 있도록 겸손한 자세로 결과를 보고한다.


## 버그 수정 절차
 1. 버그를 재현하는 테스트 코드를 먼저 작성한다.
    - Robolectric: `simple_*/src/test/java/.../robolectric/`
    - Unit: `simple_*/src/test/java/.../unit/`
 2. 테스트가 실패함을 확인한다. (버그 재현 증명)
 3. 버그를 수정한다.
 4. 테스트가 통과함을 확인한다.
 5. 기존 테스트 전체가 깨지지 않았는지 확인한다.

## 수정/작업 금지 패턴
 - 요청하지 않은 파일을 수정하지 않는다.
 - 사용자 승인 없이 코드를 수정하지 않는다.
 - 불필요한 리팩토링, 대량 포맷 변경을 하지 않는다.
 - 일부만 보고 전체를 수정하지 않는다.
 - 신규 코드 작성 시 단순성을 먼저 검토한다.
   - 단일 케이스에 sealed class / abstract class / interface를 설계하지 않는다.
   - 요청하지 않은 캐싱, 설정 가능성(configurability), 확장 포인트를 추가하지 않는다.
   - 자가 질문: "15년차 Android 개발자가 이 코드를 보면 과복잡하다고 할 것인가?" → Yes이면 단순화 후 재작성.
 - 내 변경으로 인해 새로 사용되지 않게 된 import/변수/함수는 제거한다.
 - 원래부터 존재하던 dead code는 삭제하지 않고 사용자에게 언급만 한다.
 - 변경된 모든 줄은 사용자 요청으로 직접 추적 가능해야 한다. 추적 불가한 변경이 있으면 별도 승인을 받는다.


## 판단 전 확인 규칙

기존 "불확실하면 질문하라"는 규칙은 *스스로 불확실하다고 느낄 때만* 작동하는 주관적 트리거다.
과신하면 발동조차 하지 않으므로, 아래 3개 규칙으로 트리거를 객관화한다.

1. **질문 턴 규칙**
   - 사용자가 질문하면 그 턴은 답변 전용이다. 도구를 호출하지 않고 묻는 것에만 답한 뒤 멈춘다.
   - 사용자의 감정 톤(짜증/재촉)을 의도로 추론해 답변을 건너뛰고 작업으로 넘어가지 않는다.
   - 문자 그대로의 질문에 먼저 답하고, 다음 행동은 지시를 받는다.

2. **객관적 확인 트리거**
   - 확신 여부와 무관하게 아래 범주에 해당하면 원문/사용자에게 확인한다.
     - (a) 정의된 절차의 단계에 진입하기 전
     - (b) 되돌리기 어려운 작업을 수행하기 전
     - (c) 사용자 의도 해석이 필요한 지점
   - 트리거는 "불확실함을 느낄 때"가 아니라 "이 범주에 해당할 때"다.

3. **기본 가정 뒤집기**
   - 내 이해는 확인되기 전까지 불완전하다고 가정한다.
   - "반박되기 전까지 맞다"가 아니라 "확인되기 전까지 불완전하다"를 기본값으로 둔다.
