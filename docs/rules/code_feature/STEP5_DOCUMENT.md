# Step 5. 문서 갱신

기능 변경으로 인해 업데이트가 필요한 문서를 갱신한다.

> README 갱신 기준: docs/rules/README_RULE.md

## 체크리스트

- [ ] 공개 API가 추가/변경되었는가? → README / SPEC 갱신
- [ ] 기존 API가 제거(@Deprecated)되었는가? → README 제거 안내 / SPEC 갱신
- [ ] 모듈 구조가 변경되었는가? → AGENTS.md 갱신
- [ ] 작업 유형별 기준에 따라 PRD/SPEC/IMPLEMENTATION_PLAN 갱신 여부를 판단하고 반영했는가?
- [ ] 변경 사항 요약을 사용자에게 보고했는가?

## README 갱신 대상 판단

| 작업 유형 | README 갱신 | KDoc 갱신 |
|---------|-----------|---------|
| 공개 API 추가 | 해당 기능 README | 필수 |
| 공개 API 개선 | 해당 기능 README | 변경 내용 반영 |
| 공개 API 제거(@Deprecated) | 해당 기능 README에 제거 예정 안내 | 필수 |
| 내부 구현만 변경 | 불필요 | 로직 설명이 달라진 경우에만 |

## 기능 명세 문서 갱신 대상 판단

> 기능 명세 문서 경로: `{모듈}/docs/feature/{기능명}/`

| 작업 유형 | PRD 갱신 | SPEC 갱신 | IMPLEMENTATION_PLAN |
|---------|---------|---------|-------------------|
| 추가 | 신규 작성 또는 갱신 | 신규 작성 또는 갱신 | 신규 작성 |
| 개선 | 요구사항 변경 시 갱신 | 공개 API/동작 변경 시 필요 | 갱신 불필요 |
| 제거 | 제거 예정 명시 | 제거 예정 명시 | 갱신 불필요 |

## 최종 보고 형식

```
## 기능 변경 완료 보고

### 작업 유형
- 추가 / 개선 / 제거

### 변경 파일 목록
- 파일명: 변경 내용

### 변경 전/후
- 변경 전: ...
- 변경 후: ...

### 문서 갱신
- 갱신한 문서 목록 (없으면 "없음")

### 빌드/테스트 결과
- 빌드: 성공
- 테스트: 전체 통과
- apiCheck: 통과
```

## 심각도 기준

- HIGH: 공개 API 추가/변경 후 README 미갱신
- MEDIUM: 공개 API 추가 후 KDoc 미작성
- MEDIUM: 변경 사항 요약 보고 누락
