# Step 1. 요구사항 분석 & 설계

무엇을 추가/개선/제거할 것인가? 영향 범위는 어디까지인가?

> 아키텍처/설계 원칙 기준: docs/rules/coding_rule/CODE_ARCHITECTURE.md

## 작업 유형 확인

| 유형 | 판단 기준 |
|------|---------|
| 추가 (add) | 존재하지 않는 API/기능을 새로 만든다 |
| 개선 (improve) | 기존 API/기능의 동작을 변경한다 (하위 호환 유지) |
| 제거 (remove) | 기존 API/기능을 삭제한다 (@Deprecated 선언 필수) |

## 체크리스트

- [ ] 작업 유형(추가/개선/제거)을 확인했는가?
- [ ] 해당 기능의 PRD/SPEC 문서(`{모듈}/docs/feature/{기능명}/`)를 확인했는가?
- [ ] PRD/SPEC가 없는 경우, 신규 작성 필요 여부를 사용자에게 확인했는가?
- [ ] 영향 범위(호출부, 의존 모듈, 테스트)를 파악했는가?
- [ ] 구현 계획과 대상 파일 목록을 사용자에게 공유했는가?
- [ ] 사용자 승인을 받았는가? (승인 전 코드 수정 금지)

## 유형별 추가 확인

### 추가 (add)
- 기존 API와 이름/역할 충돌이 없는가?
- 아키텍처 계층에 올바르게 배치되는가?

### 개선 (improve)
- 하위 호환을 유지할 수 있는가?
- 기존 호출부에 영향을 주는가? → 영향 범위 파악 필수

### 제거 (remove)
- @Deprecated 선언 없이 즉시 삭제 금지
- 의존하는 호출부가 모두 파악되었는가?
- 버전 정책 확인: 제거가 Breaking change인지 판단했는가?
- @Deprecated 공지 이후 실제 제거는 다음 Major 버전에서만 진행

## 심각도 기준

- HIGH: 영향 범위 파악 없이 구현 착수
- HIGH: 사용자 승인 없이 코드 수정 시작
- HIGH: 제거 시 @Deprecated 없이 즉시 삭제
- MEDIUM: PRD/SPEC 미확인 상태로 진행
