# Step 1. 대상 선정 & 영향 범위 파악

리팩토링이 왜 필요한가? 범위는 어디까지인가?

> 아키텍처/설계 원칙 기준: docs/rules/coding_rule/CODE_ARCHITECTURE.md

## 진입 분기

### agent-review 결과가 있는 경우
이미 리뷰 이슈 요약이 제공된 경우, 해당 이슈 목록을 대상으로 설정한다.
- 코드 스멜 분석은 생략하고 이슈 목록에서 영향 범위만 추가로 파악
- 체크리스트 1~2번 항목은 리뷰 결과로 대체 가능
- 대상 파일 목록과 사용자 승인(3~5번)은 반드시 수행

### agent-review 결과가 없는 경우
체크리스트 전체를 순서대로 수행한다.

## 체크리스트

- [ ] 리팩토링이 필요한 이유가 명확한가? (코드 스멜, 기술 부채, 구조 개선 등)
- [ ] 코드 스멜 유형을 식별했는가?
- [ ] 영향 범위(호출부, 의존 모듈, 테스트)를 파악했는가?
- [ ] 리팩토링 범위와 대상 파일 목록을 사용자에게 공유했는가?
- [ ] 사용자 승인을 받았는가? (승인 전 코드 수정 금지)

## 코드 스멜 유형

| 유형 | 판단 기준 |
|------|---------|
| SRP 위반 | 클래스/함수를 한 문장으로 설명할 때 "그리고"가 들어감 |
| 중복 코드 | 동일/유사 로직이 2곳 이상 존재 |
| 긴 함수 | 한 함수에서 여러 추상화 레벨이 섞임 |
| 과도한 매개변수 | 함수 파라미터가 4개 이상 |
| 잘못된 레이어 배치 | ViewModel에 비즈니스 로직, Repository에 UI 로직 등 |
| 마법 숫자/문자열 | 설명 없는 하드코딩 값 |

## 심각도 기준

- HIGH: 영향 범위를 파악하지 않고 리팩토링 착수
- HIGH: 사용자 승인 없이 코드 수정 시작
- MEDIUM: 리팩토링 이유가 불명확한 상태로 진행
