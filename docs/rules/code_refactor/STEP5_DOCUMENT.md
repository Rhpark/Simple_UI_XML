# Step 5. 문서 갱신

리팩토링으로 인해 문서 업데이트가 필요한가?

> README 갱신 기준: docs/rules/README_RULE.md

## 체크리스트

- [ ] 공개 API 시그니처가 변경되었는가? → 해당 README / SPEC 갱신
- [ ] 모듈 구조가 변경되었는가? → AGENTS.md 갱신
- [ ] 패키지/클래스명이 변경되었는가? → README 예제 코드 / SPEC 갱신
- [ ] 해당 기능의 PRD/SPEC/IMPLEMENTATION_PLAN이 존재하는가? → 갱신 대상 판단
- [ ] 변경 사항 요약을 사용자에게 보고했는가?

## README 갱신 대상 판단

| 변경 내용 | README 갱신 | KDoc 갱신 |
|---------|-----------|---------|
| 공개 API 추가/제거/변경 | 해당 기능 README | 필요 |
| 클래스명/패키지명 변경 | 해당 기능 README + README.md | 필요 |
| 내부 구현만 변경 | 불필요 | 로직 설명이 달라진 경우에만 갱신 |
| 모듈 경계 변경 | AGENTS.md + README | 필요 |

## 기능 명세 문서 갱신 대상 판단

> 기능 명세 문서 경로: `{모듈}/docs/feature/{기능명}/`

| 변경 내용 | PRD 갱신 | SPEC 갱신 | IMPLEMENTATION_PLAN |
|---------|---------|---------|-------------------|
| 공개 API 시그니처 변경 | 불필요 | 필요 | 갱신 불필요 (완료 계획) |
| 클래스명/패키지명 변경 | 불필요 | 필요 | 갱신 불필요 |
| 외부 인터페이스/제약 변경 | 필요 | 필요 | 갱신 불필요 |
| 내부 구현만 변경 | 불필요 | 불필요 | 갱신 불필요 |

> 기능 명세 문서가 존재하지 않는 경우 이 단계를 건너뜁니다.

## 최종 보고 형식

```
## 리팩토링 완료 보고

### 변경 파일 목록
- 파일명: 변경 내용

### 개선 사항
- 변경 전: ...
- 변경 후: ...

### 문서 갱신
- 갱신한 문서 목록 (없으면 "없음")

### 빌드/테스트 결과
- 빌드: 성공
- 테스트: 전체 통과
```

## 심각도 기준

- HIGH: 공개 API 변경 후 README 미갱신
- MEDIUM: 변경 사항 요약 보고 누락
