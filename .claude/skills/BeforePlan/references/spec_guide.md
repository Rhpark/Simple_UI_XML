# SPEC 작성 가이드

## 역할

SPEC은 PRD(요구사항)를 **코드를 직접 탐색하며** 기술 명세로 구체화한 문서다. "어떻게" 를 다룬다.
PRD/SPEC 모두 /plan에 전달되는 입력이다.

## 핵심 원칙: 코드 탐색 기반 작성

- PRD를 펴놓고, 관련 코드를 **직접 읽으며** SPEC을 작성한다 (추측으로 명세하지 않는다).
- 모든 기술적 주장에 **파일:라인 근거**를 단다 (변경 대상, 재사용 대상, 의존성 등).
- 탐색 범위가 넓으면 Explore SubAgent에 위임할 수 있으나, 결과는 핵심 파일을 직접 읽어 교차 검증한다.

## 작성 중 문제 발견 시 분기 (SKILL.md와 동일 기준)

| 상황 | 처리 |
| --- | --- |
| 코드와 안 맞는 기술적 불일치 (질문 불필요) | **SPEC만 수정** |
| 답이 기술 수준인 질문 | 질문 → PROMPT 추가 → **SPEC 수정** |
| 답이 요구사항을 건드림 | 질문 → PROMPT 추가 → **PRD 수정(사용자 승인)** → SPEC 수정 |

- 판단: "이게 요구사항(무엇/왜)을 바꾸나?" → 예: PRD 경로 / 아니오: SPEC 경로
- 확신 여부와 무관하게 요구사항을 건드리면 사용자에게 확인한다.

## 완료 기준

- [ ] PRD의 각 요구사항이 SPEC의 기술 항목으로 대응됨
- [ ] 공개 API 정의: 신규/변경 시그니처 명시 + apiDump 갱신 대상 표기 (있으면)
- [ ] 변경/추가 대상이 파일:라인으로 특정됨 (신규 파일은 경로 + (신규) 표기)
- [ ] 재사용 가능한 기존 코드·유틸을 명시 (파일:라인)
- [ ] 동작 명세 (해당 시): 플로우 / 비즈니스 룰 / 예외 처리 / 상태 변화 / 데이터 정의
- [ ] 영향 범위(코드·모듈) / Breaking Change 여부 판단 (사용자·기능 영향은 PRD)
- [ ] 개발환경·도구 체인 제약 확인 (docs/rules/project/DEV_ENV_RULE.md 대조)
- [ ] 테스트 전략 (PROCESS_RULE.md 기준: Unit / Robolectric)
- [ ] 코드로 미확인된 빈틈 없음

> 위 완료 기준은 SPEC 산출물의 `## 11. 완료 체크리스트` 섹션으로 **미러링**한다 (단일 출처: 본 가이드. 산출물에서 항목을 임의 추가/삭제하지 않는다).
> 산출물 체크리스트의 각 항목에는 **근거(파일:라인) 또는 "해당 없음" 사유**를 1줄 단다. **근거 없는 `[x]` 금지** (PROCESS_RULE.md: 추정으로 완료 선언 금지).

## 섹션 템플릿

```md
# SPEC

## 1. 기술적 타당성
- 요구사항별 구현 가능성 검토 (신규 API 필요 여부 등)

## 2. 공개 API 정의
- 신규/변경 공개 시그니처 (함수·클래스). 없으면 "없음".
- apiCheck 베이스라인(apiDump) 갱신 대상 표기

## 3. 변경/추가 대상
- {클래스/함수/파일} : 파일:라인 — 변경 내용 (신규는 경로 + (신규))

## 4. 재사용
- 신규 작성 전 확인 순서: ① 기존 코드/유틸 → ② Kotlin/Android 표준 API → ③ 이미 선언된 의존성. 앞 단계로 해결되면 신규 작성하지 않는다. 신규 의존성 추가는 근거 필수.
- {기존 유틸/클래스} : 파일:라인 — 어떻게 재사용

## 5. 동작 명세 (해당 시)
- 플로우: 사용자/호출 동작 흐름
- 비즈니스 룰: 조건·정책
- 예외 처리: 실패·예외 시 동작 (에러 처리 규칙: docs/rules/coding_rule/patterns/CODE_PATTERNS_EXCEPTION.md)
- 상태 변화: 상태형 기능의 전이 (상태 있으면 필수)
- 데이터 정의: 필요한 데이터 구조

## 6. 권한 / 로깅 (해당 시)
- 필요 권한 / manifest 선언
- Logx 로깅 지점

## 7. 영향 범위 / Breaking Change
- 코드·모듈 관점 (사용자·기능 영향은 PRD)

## 8. 환경·도구 제약
- compileSdk/minSdk/Kotlin/AGP 영향, SDK 분기 필요 여부

## 9. 테스트 전략
- Unit / Robolectric, 대상 시나리오

## 10. 리스크 / 미확인 항목
- ...

## 11. 완료 체크리스트
> 본 가이드 "완료 기준"의 미러. 각 항목에 `[x]/[ ]`와 근거(파일:라인) 또는 "해당 없음" 사유를 1줄 단다. 근거 없는 `[x]` 금지.
- [ ] PRD 각 요구사항이 기술 항목으로 대응됨 — 근거:
- [ ] 공개 API 정의(신규/변경 시그니처 + apiDump 대상, 없으면 "없음") — 근거:
- [ ] 변경/추가 대상이 파일:라인으로 특정됨 — 근거:
- [ ] 재사용 대상 명시(파일:라인) — 근거:
- [ ] 동작 명세(플로우/룰/예외/상태/데이터, 해당 시) — 근거:
- [ ] 영향 범위(코드·모듈)/Breaking Change 판단 — 근거:
- [ ] 환경·도구 체인 제약 확인 — 근거:
- [ ] 테스트 전략(Unit/Robolectric) — 근거:
- [ ] 코드로 미확인된 빈틈 없음 — 근거:
```

## 하지 말 것

- 코드를 안 읽고 "그럴 것이다"로 명세하지 않는다.
- 근거 없는 파일:라인을 적지 않는다.
- 요구사항 변경을 SPEC에서 임의로 처리하지 않는다 (PRD 경로로).
- 과설계 금지: 단일 케이스에 sealed/abstract/interface, 요청 없는 확장 포인트를 넣지 않는다 (PROCESS_RULE.md).
