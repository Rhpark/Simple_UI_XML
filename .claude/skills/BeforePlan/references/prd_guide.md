# PRD 작성 가이드

## 역할

PRD는 PROMPT(대화 로그)에서 도출한 **요구사항 정제본**이다. "무엇을·왜" 를 다룬다.
PROMPT와 **병행 작성**하며(대화가 진행될수록 갱신), /plan에 전달되는 입력 문서다.

## 편집 규칙

- 편집 가능한 정제본이다 (PROMPT와 달리 기존 내용을 고쳐 쓸 수 있다).
- **PRD는 사용자 발언이 뒷받침될 때만 변경된다.**
  - 사용자가 말한 내용(자발적 답변이든, 내 질문에 대한 답이든)을 반영하는 것은 **자유롭게** 한다.
    그 발언이 곧 근거이자 승인이며, 별도 승인 절차를 두지 않는다.
  - 발언이 없는데(예: 코드 탐색 중 발견) 요구사항을 바꿔야 하면, **먼저 질문**해 답을 받고 그 답을 근거로 반영한다.
  - 내가 임의로(사용자 발언 없이) 요구사항을 추가·축소·변경하지 않는다.

## 완료 기준 (종료 게이트)

PRD가 아래를 "빈틈없이" 담으면 다음 단계(SPEC/코드 탐색)로 진행한다.

- [ ] 개요: 무엇을 변경하는가
- [ ] 변경 배경: 왜 이 변경이 필요한가
- [ ] 현재 상태: 현재 어떻게 동작하는가
- [ ] 목표 상태: 변경 후 어떤 상태가 되어야 하는가
- [ ] 변경 범위: In scope / Out of scope
- [ ] 영향 범위: **사용자·기능 관점** 영향 (코드·모듈 영향은 SPEC에서)
- [ ] 제약: 환경·버전·호환성·금지 사항
- [ ] 성공 조건: 어떤 상태가 되면 완료로 보는가 (검증 가능하게)
- [ ] 리스크: 예상되는 위험
- [ ] 미해결 이슈: 남은 질문이 있으면 명시 (있으면 아직 미완료)

> 위 완료 기준은 PRD 산출물의 `## 11. 완료 체크리스트` 섹션으로 **미러링**한다 (단일 출처: 본 가이드. 산출물에서 항목을 임의 추가/삭제하지 않는다).
> 산출물 체크리스트의 각 항목에는 **근거(사용자 발언/PROMPT 위치) 또는 "해당 없음" 사유**를 1줄 단다. **근거 없는 `[x]` 금지** (PROCESS_RULE.md: 추정으로 완료 선언 금지).

## 섹션 템플릿

```md
# PRD

**상태**: 작성중 / 확정

## 1. 개요
- 무엇을 변경하는가

## 2. 변경 배경
- 왜 이 변경이 필요한가

## 3. 현재 상태
- 현재 어떻게 동작하는가

## 4. 목표 상태
- 변경 후 어떤 상태가 되어야 하는가

## 5. 변경 범위
- In scope: ...
- Out of scope: ...

## 6. 영향 범위 (사용자·기능 관점)
- 어떤 사용자 흐름/기능이 영향받는가  (코드·모듈 영향은 SPEC에서 다룸)

## 7. 제약
- 환경·버전·호환성·금지 사항

## 8. 성공 조건
- 어떤 상태가 되면 완료로 보는가 (검증 가능하게)

## 9. 리스크
- 예상되는 위험

## 10. 미해결 이슈
- 남은 질문/결정 대기 (있으면 PRD 미완)

## 11. 완료 체크리스트
> 본 가이드 "완료 기준"의 미러. 각 항목에 `[x]/[ ]`와 근거(사용자 발언/PROMPT 위치) 또는 "해당 없음" 사유를 1줄 단다. 근거 없는 `[x]` 금지.
- [ ] 개요 — 근거:
- [ ] 변경 배경 — 근거:
- [ ] 현재 상태 — 근거:
- [ ] 목표 상태 — 근거:
- [ ] 변경 범위(In/Out scope) — 근거:
- [ ] 영향 범위(사용자·기능) — 근거:
- [ ] 제약 — 근거:
- [ ] 성공 조건(검증 가능) — 근거:
- [ ] 리스크 — 근거:
- [ ] 미해결 이슈 명시(없으면 "없음") — 근거:
```

## 하지 말 것

- 사용자 승인 없이 요구사항을 바꾸지 않는다.
- 기술 구현 방법(어떻게)을 PRD에 적지 않는다 → 그것은 SPEC.
- 추측을 확정 요구사항으로 적지 않는다 (모호하면 질문하고 PROMPT에 기록).
